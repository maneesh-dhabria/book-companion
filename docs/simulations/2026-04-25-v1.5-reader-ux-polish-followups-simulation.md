  - SSE event payload changes (existing summarization SSE contract is reused unchanged)
- **Companion specs:** none (parent v1.5 spec is referenced for D9 / D12 context but not changed)

## 6. Targeted Pseudocode

### 6.1 Flow A — `applyPreset(N)` with corner cases

**Entry:** PresetCards `@click` on a system card; or any caller that wants to set the applied preset to a system one.

```
FUNCTION applyPreset(presetId: number) -> void:
  preset = presets.value.find(p => p.id === presetId)
  IF preset IS undefined:
    console.warn("applyPreset: unknown id", presetId)
    RETURN                                     # FR-F4.8a silent no-op

  appliedPresetKey.value = `system:${presetId}`
  applySettingsFromPreset(preset)              # mutates currentSettings ref
  pendingCustom.value = null                   # FR-F4.10
  editingCustom.value = false                  # FR-F4.10
  IF colorPickerOpen.value:
    closeColorPicker()                         # FR-F4.10

  TRY:
    localStorage.setItem('bookcompanion.reader-applied.v1', appliedPresetKey.value)
  CATCH e:                                     # FR-F4.7b
    IF NOT lsErrorToastFiredThisSession:
      useUiStore().showToast(
        "Couldn't save your reader preferences — browser storage may be unavailable.",
        'error', 8000)
      lsErrorToastFiredThisSession = true
```

- **DB calls:** none — fully client-side.
- **State transitions:**
  - `appliedPresetKey`: `null` | `system:M` | `custom` → `system:N`
  - `editingCustom`: any → `false`
  - `pendingCustom`: any → `null`
  - color picker (if open): open → closed
- **Error branches:**
  - Unknown preset ID → silent no-op + `console.warn` (FR-F4.8a)
  - localStorage `setItem` throws → one-shot error toast (FR-F4.7b); in-memory state still applies; user sees their selection until reload
- **Concurrency notes:**
  - No cross-tab sync. Two tabs writing the same key: last writer wins on next reload (G4 documented).
  - No advisory lock needed — single-user, single-device assumption (workstream rule 2026-04-24).

### 6.2 Flow B — `PATCH /api/v1/settings` (all-or-nothing)

**Entry:** HTTP `PATCH /api/v1/settings` with partial nested body; or `bookcompanion config set <key> <value>` via shared service.

```
ENDPOINT PATCH /api/v1/settings(body: dict):
  current = settings_service.load()             # current YAML dump as dict
  candidate = deep_merge(current, body)         # body wins on overlapping keys

  TRY:
    Settings.model_validate(candidate)          # raises pydantic.ValidationError
  EXCEPT ValidationError as e:
    RETURN 400 with {"detail": e.errors()}      # YAML unchanged

  # Validation passed — persist atomically
  settings_service.write_yaml_atomic(candidate) # temp file + os.rename
  settings_service.reload_in_memory()           # refresh process-wide Settings instance
  RETURN 200 with get_safe_settings()
```

- **DB calls:** none — filesystem only. (`settings.yaml` under XDG config dir, atomic temp+rename.)
- **State transitions:**
  - YAML on disk: `prior` → `candidate` (atomic single-step rename — no partial-write window)
  - Process-wide `Settings` instance: `prior` → `reloaded`
- **Error branches:**
  - Pydantic validation fails → 400; YAML unchanged; process settings unchanged. Frontend toasts the first error message.
  - Filesystem write fails (permission, disk full) → 500 (existing behavior); YAML unchanged due to atomic rename never replacing the original.
  - Network drop after backend persists but before client receives 200 → client retries with same body → no harm (idempotent — see G6 patch in §7.1).
- **Concurrency notes:**
  - Two simultaneous PATCHes from different tabs/clients: serialized by the atomic temp+rename. Last writer wins. No advisory lock; no merge — documented in §7.1 post-G3 patch.
  - The CLI surface (`bookcompanion config set`) calls the same `update_settings()` and inherits the same atomicity (FR-F1.8 + D18).
