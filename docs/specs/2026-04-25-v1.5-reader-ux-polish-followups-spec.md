## 1. Problem Statement

Hands-on testing of the v1.5 reader UX implementation surfaced seven defects spanning settings persistence, missing UI wiring per the v1.5 spec, redundant tag-input affordance, a reader-settings state-machine tangle, and search-palette jank. Beneath the named bugs is one structural problem — **reader presets have three competing sources of truth** (server-persisted user presets, browser-local Custom slot, server-tracked active flag) that produce inconsistent applied state when the user crosses them.

This spec collapses the three sources of truth to **one client-side `appliedPresetKey: string`**, removes server-persisted user presets entirely (clean break — table becomes a system-presets read store), wires the v1.5-promised Summarize button onto the new overview page, fixes the silent settings-save round-trip, refactors the tag-add affordance, and lands a global toast outlet so no save in this scope can fail silently.

**Primary success metric:** every reproducer in the requirements doc passes manual verification on the Gutenberg fixture book without regression in any v1.5-shipped feature.

## 3. Non-Goals

- **Not** preserving existing user-created server-side presets — because the goal is a clean break and the dev DB is empty (verified: `SELECT count(*) FROM reading_presets` = 0).
- **Not** touching the `LLMConfig` fields beyond `config_dir` — because other fields (provider, model, timeout, etc.) work today.
- **Not** redesigning the reader settings popover layout — because the visual layout is fine; only the state machine and the Custom-card affordance change.
- **Not** changing tag autocomplete / suggestion behavior in `TagChipInput` — because only the entry-affordance is in scope.
- **Not** changing what the search palette searches across — because only the debounce and first-key reliability change.
- **Not** building a cross-device theme sync — because the workstream-level decision (2026-04-24) is "one Custom slot per device", reaffirmed here.
- **Not** revisiting D12 (no summarize button in reader header) — because that decision stands; the gap is missing overview wiring.
- **Not** validating `LLMConfig.cli_command` for shell-injection or path-existence at PATCH time — because the user runs the binary themselves; the worst case is a clear "command not found" stderr surfaced by the existing summarize error path.

## 5. User Journeys

The five journeys from the requirements doc are authoritative; this spec adds the technical contract that makes each one work. Re-stated briefly:

1. **Custom CLI binary** — Settings → LLM Provider → edit `CLI command` → Save → success toast → reload → value persists.
2. **Resume summarization from overview** — `/books/:id` → see `K failed · Retry failed` → click → spinner + disabled → SSE drives state to `Summarize 7 of 17 pending` → eventually `✓ All 17 summarized`.
3. **Tagging a section** — Reader header → `+ Add tag` button → click → focused input → Enter → tag chip appears, button returns. Esc/blur-empty cancels.
4. **Custom theme then back to system** — Reader settings → Custom card (seeded from active system) → Save & Apply → applied. Click Sepia → applied; pencil icon disappears from Custom. Reload → Sepia preserved. Click Custom → previous Custom values restored.
5. **Quick search** — ⌘K → first keystroke captured → 300ms pause → results appear.

## 7. API Changes

### 7.1 `PATCH /api/v1/settings` — strict validation

**Request:** unchanged shape (partial nested object).

**Behavior change:**
- Unknown top-level keys, unknown nested keys, or values failing field validation return **400**.
- All-or-nothing: any invalid field in the body causes the entire PATCH to be rejected; the on-disk YAML is unchanged on a 400.

**Response (200):** unchanged shape, returned by `get_safe_settings()`. Now includes `cli_command` (FR-F1.3); no longer includes `config_dir` (FR-F1.2).

**Response (400):**
```json
{
  "detail": [
    { "loc": ["llm", "timeout_seconds"], "msg": "Input should be a valid integer", "type": "int_parsing" }
  ]
}
```

The shape matches FastAPI's standard pydantic validation error response so the existing client error handler ingests it without a custom branch. The frontend renders the first `msg` in a toast.

### 7.2 `GET /api/v1/reading-presets`

**Response (200), updated shape:**
```json
{
  "items": [
    {
      "id": 1,
      "name": "Light",
      "font_family": "Georgia",
      "font_size_px": 16,
      "line_spacing": 1.6,
      "content_width_px": 720,
      "theme": "light",
      "created_at": "2026-04-24T10:00:00Z"
    }
  ]
}
```

Removed fields: `default_id` (response wrapper), `is_system`, `is_active` (per item).

### 7.3 Removed Endpoints

- `POST /api/v1/reading-presets`
- `PATCH /api/v1/reading-presets/{preset_id}`
- `DELETE /api/v1/reading-presets/{preset_id}`
- `POST /api/v1/reading-presets/{preset_id}/activate`

All return 404 (FastAPI default) once the route handlers and their `router.add_api_route` registrations are removed.

### 8.2 State Management

- **`appliedPresetKey: Ref<string \| null>`** in `readerSettings` store, persisted under localStorage key `bookcompanion.reader-applied.v1`. Single source of truth for "currently applied" reader preset.
- **`customTheme: Ref<CustomThemeSlot | null>`** unchanged from v1.5; persisted under `bookcompanion.reader-custom.v1`.
- **`presets: Ref<ReadingPreset[]>`** — system-only after migration; loaded once on store init and on app focus.
- **`pendingCustom`, `editingCustom`, `dirty`** — unchanged shape, but `pendingCustom` is forcibly cleared whenever `applyPreset()` is called (FR-F4.10).
- **`useUiStore().toasts`** — array of active toasts, FIFO; `showToast()` pushes; `dismissToast(id)` removes; auto-dismiss timers managed inside the store.

### 8.3 UI Specifications

#### TagChipInput — button → input swap
- **Button mode:** `<button>` element, no border, no background, `text-sm`, color `gray-500` rest / `gray-700` hover, label `+ Add tag`. Sits at the end of the chip row, vertically aligned with chip baselines.
- **Input mode:** `<input type="text">`, no border, no background, `border-bottom: 1px solid gray-300` underline only, font matches surrounding chips, width matches the button slot it replaced (no layout reflow on transition).
- **Transition:** synchronous `editing` toggle; focus on the input via `nextTick`. Reverse transition on Esc / blur-empty.

#### SummarizationProgress placement on overview
- Container: existing action area in `BookOverviewView.vue` (currently lines 51-62 per recon). Layout: flex row `gap-2`, `<RouterLink>Read</RouterLink>` on the left, `<SummarizationProgress>` on the right.
- All three states render as a single button-shaped element; no inline status text outside the component.

#### Custom card pencil icon
- Position: absolute `top-1 right-1` of the card.
- Shape: 16×16 px pencil glyph (Heroicons / Lucide — pick whichever the project already uses).
- Visibility: `v-if="store.appliedPresetKey === 'custom'"`.
- Click: `@click.stop="openCustomPicker()"` — `.stop` prevents the card's own click handler from firing.

#### ToastContainer
- Position: fixed top-right, `z-50`, stacked vertically with 8px gap between toasts.
- Per toast: rounded card, colored left border per type (green/red/amber/blue), short message (max 2 lines), close button. Slide-in from right + fade-out on dismiss.
- Auto-dismiss: per-toast `duration` ms; manual close via the × button.

## 9. Edge Cases

| # | Scenario | Condition | Expected Behavior |
|---|----------|-----------|-------------------|
| E1 | LLM settings: empty `cli_command` saved | User clears the field, clicks Save | Backend pydantic accepts empty string (string field, no minLength); summarize attempts will fail at `shutil.which("")` returning None, surfacing the existing "no provider detected" banner. (Spec choice: do **not** add a minLength to avoid coupling the schema to UI behavior.) |
| E2 | LLM settings: PATCH with unknown nested key | Body contains `{"llm": {"foo": "bar"}}` | 400 with pydantic error in `detail`; frontend renders error toast `"Settings save failed: foo is not a valid field"` (or the raw pydantic message if simpler). |
| E2a | LLM settings: PATCH body has both valid and invalid fields | Body contains `{"llm": {"cli_command": "claude-personal", "timeout_seconds": "bad"}}` | 400; **neither** field is persisted (D17 all-or-nothing); the YAML on disk is unchanged; frontend toast carries the validation error for `timeout_seconds`. |
| E3 | Migration runs on a DB with non-empty user presets | A pre-release user has saved presets | All `is_system=false` rows are deleted in the same transaction as the column drop. User loses those presets; if any was the active preset (`appliedPresetKey` references a now-missing ID after upgrade), the store falls back to the first available system preset on next load (FR-F4.11). |
| E4 | `appliedPresetKey` references a system preset ID that no longer exists | System preset list shrinks (future change) | On `loadPresets()`, if `appliedPresetKey === "system:N"` and `presets.find(p => p.id === N)` is undefined, set `appliedPresetKey = "system:{firstAvailableId}"` and apply that preset's settings. No error toast (silent recovery). |
| E5 | User clicks Custom card while Custom slot is empty (first-edit) | `customTheme === null && appliedPresetKey !== "custom"` | Seed `customTheme` from the currently applied system preset's values (or Light if `appliedPresetKey === null`); set `appliedPresetKey = "custom"`; open the color picker. |
| E6 | User clicks system preset while in the middle of editing Custom (color picker open) | `editingCustom === true` and any system card clicked | `pendingCustom` is discarded silently; `appliedPresetKey` flips to `system:{N}`; picker closes; checkmark moves. No confirmation prompt. |
| E7 | Tag input: user types, then opens command palette via ⌘K | Tag input is `editing=true`; ⌘K fires | Command palette captures focus; tag input blurs; if non-empty, the tag is committed (FR-F3.5); if empty, the input dismisses. No tag is lost silently. |
| E8 | Search palette: user opens, types, closes, reopens | Two open events with input between | Reopen starts blank (FR-F5.5); previous query is not preserved across opens. |
| E9 | Toast: many toasts queued | User triggers 5 saves rapidly | Each appears in the stack; newest at the top; oldest auto-dismisses first. No upper limit enforced (toasts max ~5 in practice for this scope). |
| E10 | Summarize button: POST starts but SSE never fires | Job ID returned; no events; 30s elapses | Existing `summarizationJobStore` polling fallback (per parent v1.5) kicks in; button stays disabled with spinner until polled status updates the count. |
| E11 | BookOverviewView: book has 0 sections | `total === 0` | Render `✓ All 0 summarized` (non-interactive) — or render nothing. **Decision:** render nothing — a 0-section book is degenerate and the empty-state copy elsewhere already handles it. |
