## Overview

This plan executes the seven-defect cleanup from the v1.5 polish-followups spec across backend and frontend. The structural change is collapsing reader-preset state from three sources of truth (server `is_active`, server user-presets, browser Custom slot) to one client-side `appliedPresetKey` string, with a forward-only Alembic migration that captures any legacy active preset name into a sidecar file the frontend reads once on first launch. Five smaller fixes ride along: strict pydantic validation on `PATCH /api/v1/settings` with shared CLI parity, a global toast outlet in `AppShell`, the `SummarizationProgress` button moved from the reader header to the book overview, a `+ Add tag` button-to-input swap on `TagChipInput`, and `v-show` pre-mounting + `useDebounceFn(300)` on `CommandPalette`. Backend lands first so the frontend never points at code that hasn't shipped, then the global toast is wired before any UI surface that depends on it.

**Done when:** every reproducer in §5 of the spec passes manual verification on the Gutenberg fixture book; backend test suite is green (no `integration_llm`); `npm run type-check && npm run build && npm run test:unit` all pass; `ruff check`/`ruff format --check` are clean on changed files; the migration applies forward-only against a baseline DB seeded with one legacy `is_active=true` user preset and the legacy hint round-trips into `appliedPresetKey` on first frontend reload; no `config_dir` references remain anywhere under `backend/app/` or `frontend/src/`.

## Decision Log

> Inherits all 18 decisions (D1-D18) from the spec. Entries below are implementation choices made during planning.

| # | Decision | Options Considered | Rationale |
|---|----------|-------------------|-----------|
| P1 | Strict PATCH validation is implemented by adding `model_config = ConfigDict(extra="forbid")` to **every nested config sub-model** (`LLMConfig`, `NetworkConfig`, `WebConfig`, `BackupConfig`, `SummarizationConfig`, `EmbeddingConfig`, `SearchConfig`, `ImageConfig`, `StorageConfig`, `LoggingConfig`, `DataConfig`, `DatabaseConfig`) but **NOT** the root `Settings` model. `update_settings()` deep-merges the patch into `self.settings.model_dump()` and re-validates via `Settings.model_validate(merged)` — pydantic raises `ValidationError` on unknown nested keys, type mismatches, or invalid enum values. | (a) `extra="forbid"` only on root, (b) `extra="forbid"` on nested only, (c) hand-rolled per-field whitelist | (b). Forbidding extras on the root would break adding new top-level config sections; forbidding them on nested gives the spec's "unknown nested keys → 400" without locking out future top-level extension. Re-validating the merged dict gives all-or-nothing semantics for free (D17). |
| P2 | Settings PATCH translates `pydantic.ValidationError` → `fastapi.HTTPException(400, detail=err.errors())`. The `errors()` shape (`[{"loc":[...], "msg":"...", "type":"..."}]`) matches FastAPI's standard validation envelope, so the existing client error handler (`api/client.ts:handleResponse`) ingests it via `body.detail` without a custom branch. | (a) FastAPI shape, (b) custom shape, (c) only first error | (a). Spec §7.1 explicitly calls for FastAPI's standard shape. |
| P3 | Legacy-active capture (FR-F4.1 step 1) is stored as a **sidecar text file** at `{data.directory}/migration_v1_5_1_legacy_active.txt` containing the captured preset name (UTF-8, single line). A new endpoint `GET /api/v1/reading-presets/legacy-active-hint` returns `{name: string \| null}` if the file exists, and **deletes the file on first successful read**. | (a) sidecar file + GET endpoint, (b) `_migration_state` row, (c) settings YAML key | (a). The data directory already houses similar one-shot artefacts (model cache, library.db); adding a settings YAML key would persist a transient migration concern into user-visible config; a `_migration_state` table is heavier than warranted for a single-string one-shot read. The "delete on first read" rule satisfies the "consumed exactly once per device" clause of FR-F4.7c. |
| P4 | The six v1.5 system presets seeded by FR-F4.5 are: **Light, Sepia, Dark, Night, Paper, High Contrast** (matches the `shippedThemes` already in `ReaderSettingsPopover.vue:26-33`). Font/size/spacing/width default to Georgia / 16 / 1.6 / 720 (matches `DEFAULT_SETTINGS` in `readerSettings.ts:19-25`); `theme` column gets the kebab-case theme id (`light`, `sepia`, `dark`, `night`, `paper`, `contrast`). | (a) reuse shippedThemes names, (b) introduce a different curated list | (a). The frontend already renders these six in the theme picker; using the same names for the DB-backed `presets[]` keeps the two surfaces aligned and avoids a UX mismatch. |
| P5 | The system-preset seed runs **inside the same Alembic migration** that drops the columns (after the column drop, before the migration commits). It uses `op.bulk_insert` with idempotency guarded by a pre-check (`SELECT name FROM reading_presets WHERE name IN (...)`). The migration is also re-runnable without harm because the unique constraint on `name` already prevents duplicates. `init_cmd._run_migrations()` requires no change. | (a) seed in migration, (b) seed in init_cmd boot path, (c) seed in both | (a). One source of truth; the migration is the authoritative point at which "fresh DB now has 6 system presets" becomes true. Boot seeding is unnecessary because every install has run migrations. |
| P6 | The existing `backend/tests/integration/test_api/test_reading_presets_api.py` is **deleted entirely** and replaced by three new files: `test_reading_presets_routes_removed.py`, `test_reading_presets_get_shape.py`, and `test_migration_v1_5_1.py`. The corresponding pydantic schemas `ReadingPresetCreateRequest`, `ReadingPresetUpdateRequest` are deleted from `app/api/schemas.py`; `ReadingPresetResponse` and `ReadingPresetListResponse` are simplified (drop `is_system`, `is_active`, `default_id`). | (a) delete + replace, (b) edit in place, (c) keep alongside new tests | (a). The old file's 7 tests assert CRUD/activate behaviour that no longer exists; in-place editing would leave a graveyard of skipped/xfail tests. Clean delete + 3 narrow files keeps the diff reviewable. |
| P7 | `appliedPresetKey` localStorage validation regex is **`/^(system:\d+|custom)$/`** — slash-delimited. Invalid values trigger `localStorage.removeItem(key)` + `console.warn` and the store proceeds with `null`. | (a) explicit regex, (b) catch-and-default in JSON.parse, (c) full schema parse | (a). Spec FR-F4.7a names this contract; the regex is the cheapest enforcement. The value is a flat string, not JSON, so JSON.parse is irrelevant. |
| P8 | The "once per session" toast for localStorage write failure (FR-F4.7b) uses a **module-level boolean flag** (`let _quotaToastFired = false`) inside `readerSettings.ts`, NOT a Pinia ref or composable. Cleared on hard reload (i.e. once per page load), which matches the user's mental model of "session". | (a) module flag, (b) Pinia ref, (c) sessionStorage flag | (a). Module flag is the simplest implementation; a Pinia ref would survive HMR in dev but not real reloads, and sessionStorage adds a write to the very subsystem that's failing. |
| P9 | Toast container max-visible cap (FR-F6.1) is enforced **in the store**, not the component. `showToast()` checks `toasts.value.length >= 5` and pre-emptively dismisses `toasts.value[0]` (oldest) before pushing the new toast. | (a) store-side cap, (b) component-side `v-for` slice, (c) defer overflow toasts | (a). Store-side keeps the cap consistent across any future component that might render the same toast list (e.g. accessibility region). Component-side slicing would leave the dropped toast in the array, leaking timers. |
| P10 | TagChipInput's existing autocomplete + suggestions list is **preserved verbatim** — the refactor only changes the default render (button vs persistent input). When in `editing=true` mode, the existing `runSuggest` debounce, `<ul.suggest-list>` panel, and `mousedown.prevent="commit(s)"` selection all stay. | (a) preserve suggest, (b) drop suggest, (c) make suggest opt-in | (a). Spec §3 non-goal explicitly says "Not changing tag autocomplete / suggestion behavior". |
| P11 | The "starting" state for `SummarizationProgress` (FR-F2.5) is implemented with a new local `starting` ref that flips true on click of `onClick()` until the awaited `job.startJob()` resolves OR the first SSE event lands (`job.activeJobSectionId !== null`). The button label preserves the prior text but swaps the click target for an inline 14×14 spinner (same CSS as `.palette-loading .spinner`). | (a) local ref + watcher, (b) hoist to job store, (c) optimistic | (a). The state is component-local; promoting to the job store would expose it to BookOverviewView listeners that don't care. |
| P12 | TagChipInput's 64-character cap (FR-F3.8) is enforced inside `commit()`: `name = raw.trim().replace(/\s+/g, ' ')` then `if (name.length > 64) { name = name.slice(0,64); useUiStore().showToast('Tag truncated to 64 characters', 'warning'); }`. The toast fires once per commit (no extra debounce needed because Enter is the only trigger). | (a) inline cap + toast, (b) `<input maxlength="64">`, (c) backend enforcement | (a). `maxlength` would silently swallow the extra chars without telling the user; spec FR-F3.8 demands the warning toast. Backend `Tag.name` column is `String(200)` and stays — server side just won't see >64-char names because the client caps. |
| P13 | The migration's "capture legacy active name" step uses **raw SQL** (`SELECT name FROM reading_presets WHERE is_active = 1 LIMIT 1`) inside a `with op.get_context().begin_transaction():` block. The captured name is written to the sidecar file via `pathlib.Path(...).write_text(name)`. The migration **does not** depend on the SQLAlchemy ORM (the model is being changed in the same release). | (a) raw SQL + Path.write_text, (b) ORM via session, (c) Alembic Inspector | (a). Standard Alembic pattern when the live model has diverged from the migration's expected schema. Path resolution uses `Settings().data.directory` loaded at the top of the migration file. |

## Code Study Notes

**Backend impact surfaces (verified read):**

- `backend/app/config.py:21-33` — `LLMConfig` defines `config_dir: str | None = None`. Removing the field cascades to 5 caller sites (`cli/deps.py:148`, `api/deps.py:52`, `api/routes/book_summary.py:146`, `api/routes/processing.py:167`, both `summarizer/claude_cli.py:83-95` and `summarizer/codex_cli.py:29-35`). Pydantic-settings handles env var unbinding for free once the field is removed.
- `backend/app/services/settings_service.py:117-183` — `get_safe_settings()` does NOT currently expose `cli_command`; that's the visible F1 bug. `update_settings()` uses `dict.update()` semantics with no validation — root cause of unknown-key silent merge. Two write paths exist (`update_settings()` writes `DEFAULT_CONFIG_PATH = ~/.config/bookcompanion/settings.yaml`; `persist_patch()` writes to platformdirs-resolved settings.yaml — same path, redundant).
- `backend/app/cli/commands/config_cmd.py:56-73` — CLI `config set` writes to `~/.config/bookcompanion/config.yaml` (note: NOT settings.yaml!), bypassing both the service AND validation. D18 / FR-F1.8 forces this onto the same service path.
- `backend/app/api/routes/reading_presets.py` — full CRUD + activate, all 4 mutation routes need deletion. The `GET` route's `is_system.desc()` ordering and `default_id` derivation both need to go.
- `backend/app/db/models.py:533-549` — `ReadingPreset` has `is_system` and `is_active` columns, both with `server_default="false"`.
- `backend/app/api/schemas.py:443-478` — 4 pydantic schemas; 2 (`ReadingPresetResponse`, `ReadingPresetListResponse`) need field trimming, 2 (`ReadingPresetCreateRequest`, `ReadingPresetUpdateRequest`) deleted.
- Migration head verified via `alembic heads`: **`e1f2a3b4c5d6`** (v1_5c_annotation_prefix_suffix). New revision must declare `down_revision = "e1f2a3b4c5d6"`.
- **No existing seed of system presets** — verified by `grep -ln 'ReadingPreset' app/` returning only models.py + schemas.py + the route file. Spec FR-F4.5 is the first seed. Dev DB has 0 rows (matches spec line 36 verification).
- `backend/app/api/routes/books.py:101-106` — backend already exposes `summary_progress = {"summarizable", "summarized", "failed_and_pending", "pending"}` on `GET /api/v1/books/:id`. `SummarizationProgress` component expects `:total = summarizable`. No new backend endpoint needed for F2.

**Frontend impact surfaces (verified read):**

- `frontend/src/components/app/AppShell.vue:1-23` already mounts `<CommandPalette />` once inside the shell wrapper — FR-F5.1 just requires switching the inner overlay from `v-if` to `v-show` (which lives in `CommandPalette.vue:68`).
- `frontend/src/stores/ui.ts` — `useUiStore` already exposes `showToast(message, type, duration)` with `setTimeout`-based auto-dismissal; no global outlet exists. Adding the `<ToastContainer>` is net-new component work.
- `frontend/src/composables/useDebounceFn.ts` — composable exists with `cancel()` + `flush()` and auto-cancel on unmount; `CommandPalette.vue:13,22-28` currently uses a hand-rolled `setTimeout` that needs replacement.
- `frontend/src/components/common/TagChipInput.vue` — single component used at 2 callsites (`BookOverviewView.vue:32-36`, `SectionTagRow.vue:11-16`). Existing `useDebounceFn(runSuggest, 150)` autocomplete stays. Default render is currently a persistent `<input>`; needs to become a button-by-default.
- `frontend/src/stores/readerSettings.ts:94-266` — current store has `activePreset: ReadingPreset | null` + `customTheme: CustomThemeSlot | null` + `applyPreset()` calls server `activatePreset()`. The store also exports `saveAsPreset` + `deletePreset` which need removal alongside their API import.
- `frontend/src/components/settings/PresetCards.vue` filters `presets[]` by `p.is_system` to split into system + user grids; both filters disappear after the column drop. Component must render Custom alongside system in one grid (FR-F4.13/14).
- `frontend/src/components/settings/ReaderSettingsPopover.vue:97-283` — Save-as-Preset block (lines 257-269) and a standalone Custom theme card (lines 123-137) both removed; the Custom card moves into `<PresetCards>`.
- `frontend/src/components/book/SummarizationProgress.vue` already has the three states and accepts `:failed-and-pending`. `BookDetailView.vue:191-196` mounts it with only `:summarized + :total` (the bug); after this plan it's simply removed there. `BookOverviewView.vue:51-62` is the new home.

**Project conventions confirmed:**

- All migrations use `render_as_batch=True` for SQLite ALTER TABLE compatibility.
- `selectinload()` required on any repo query whose objects survive a commit; not relevant here (no new repo queries).
- Import style: `from app.db.models import ReadingPreset` (absolute from `app` root).
- Vue tests live in `__tests__/` next to the source; backend tests in `tests/unit/` and `tests/integration/test_api/`.
- Pre-existing lint baseline: `ruff check .` against `main` is clean (verified one-shot before plan).

## Risks

| Risk | Likelihood | Mitigation |
|------|-----------|------------|
| `extra="forbid"` on `LLMConfig` rejects YAML configs that legitimately contain `config_dir` from earlier installs (since the field is being removed) | Medium | T1 explicitly tests that an existing YAML containing `config_dir` is migrated by stripping the key on first read. The user's pre-release-only audience means any `config_dir` in YAML is dev-machine-only; we document the breaking change in `docs/changelog.md` per FR-F1.7. |
| Migration fails partway: legacy hint written but column drop fails | Low | The migration runs all four steps inside a single `op.execute(BEGIN)` ... `COMMIT` block (or `with op.batch_alter_table()` which is implicitly transactional in Alembic SQLite mode). If the column drop fails, the row delete + hint write roll back together. Re-runnable. |
| Sidecar file path conflicts with another tool / file | Low | Filename `migration_v1_5_1_legacy_active.txt` is namespace-distinct; the data directory is single-purpose. |
| `appliedPresetKey` localStorage write under quota pressure on private browsing leaves the in-memory state diverged from persisted state | Medium | FR-F4.7b's "once per session" toast tells the user; on next reload `appliedPresetKey` reverts to `null`. The user re-clicks. Acceptable. |
| `useDebounceFn(300)` change visibly slows the palette compared to the current 200ms; users complain | Low | Spec D12 explicitly calls for 300; if needed, the constant is one line to tune. Document the choice in the test. |
| Removing `<SummarizationProgress>` from `BookDetailView.vue` breaks any test that asserted its presence on the reader page | Medium | T17 includes a grep for `summary-progress` in `frontend/src/views/__tests__` and removes any stale assertions in the same task. |
| Tag truncation toast spams when a user pastes a long tag; warning fires once per Enter so debouncing is unnecessary | Low | Spec FR-F3.8 explicitly says one-shot per truncation; `commit()` is the only trigger and only fires on Enter / blur-with-content. |
| The frontend's `appliedPresetKey="system:{id}"` references a numeric ID that's stable only within one install; cross-device sync could mismatch | Low | Spec §3 non-goal: cross-device theme sync is out of scope. Documented. |
| Migration deletes user-created presets the user actually wanted (despite spec saying they're empty) | Low | Spec §3 non-goal explicitly forbids preservation; D3 + dev-DB verification cover this. The migration is forward-only by design. |

## Rollback

- **If migration fails partway:** `alembic -c app/migrations/alembic.ini downgrade e1f2a3b4c5d6` (the new migration is forward-only, but `downgrade` still rolls back the failed transaction's residue if any auto-commit happened). Then drop the new revision file and resolve the cause.
- **If frontend ships before backend on a given user's device:** the new client calls `GET /api/v1/reading-presets` and gets the old shape with `default_id`; the rewritten store's loader gracefully ignores extra response fields (no schema validation is performed on the GET response). Acceptable but minimise the window — backend tasks land first.
- **If `appliedPresetKey` corruption blocks the user:** advise `localStorage.removeItem('bookcompanion.reader-applied.v1')` in the browser console; the store rehydrates to `null` and Light visual defaults already apply.
- **If the legacy-active sidecar was consumed but the chosen preset name doesn't exist on first launch:** `appliedPresetKey` stays `null`; user picks Light manually. Sidecar is one-shot-per-device by design.
