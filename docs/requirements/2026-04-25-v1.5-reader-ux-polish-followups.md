# V1.5 Reader UX Polish Followups — Requirements

**Date:** 2026-04-25
**Status:** Draft
**Tier:** 2 — Enhancement / Bug Pass
**Parent v1.5 doc:** [`docs/requirements/2026-04-24-v1.5-reader-ux-and-features.md`](./2026-04-24-v1.5-reader-ux-and-features.md)
**Parent v1.5 plan:** [`docs/plans/2026-04-24-v1.5-reader-ux-and-features-implementation-plan.md`](../plans/2026-04-24-v1.5-reader-ux-and-features-implementation-plan.md)

## Problem

Hands-on testing of the v1.5 reader UX implementation surfaced **seven issues** ranging from a settings save bug that silently discards user input, a spec-mandated UI element that was never wired up, to a tangle in reader-settings state where applying a system preset after defining a Custom one is a no-op. Search-as-you-type in the command palette also drops first keystrokes and feels jittery. Together these prevent v1.5 from being a clean polish ship.

The thread tying most of them together is **competing sources of truth in reader settings** (server-persisted user presets vs. browser-local Custom slot vs. system presets) and **incomplete wiring of the v1.5 plan's overview-page work**. The fix is mostly polish + one targeted state-machine simplification, not a redesign.

### Who & Why Now

A solo user (the developer + small early-share circle) running `bookcompanion serve` and walking through the just-shipped v1.5 paths. The bugs are reproducible on the current `main`. **Now** because v1.5 was just merged and the same subsystems are still hot — bundling these followups before any further reader work avoids re-paging context.

## Goals & Non-Goals

### Goals

- **G1** — LLM Provider settings save round-trips correctly for non-default values (custom CLI binary names like `claude-personal`).
- **G2** — The book-level "Summarize N of M pending" button lives on `BookOverviewView` with the three states the v1.5 spec already defined; reader header stays clean per spec D12.
- **G3** — "Add tag" affordance on every site reads as a button, not as an empty input box.
- **G4** — The Custom preset is a single browser-local slot with no naming UX; clicking any system preset always wins, regardless of prior custom state.
- **G5** — Search-as-you-type in the command palette accepts the first keystroke reliably and debounces at 300 ms.
- **G6** — No interactive surface in this scope fails silently; every save / apply / commit produces a visible state change or explicit error message.

### Non-Goals

- **NOT** redesigning the reader settings popover layout — because the visual layout is fine; only the state machine and one save-flow are broken.
- **NOT** preserving existing user-created server-side presets — because the v1.5 design split into two redundant persistence paths and the user has chosen to consolidate to one (browser-local) slot. Any rows in `reading_presets` for `is_system=false` get dropped via migration.
- **NOT** moving any settings out of YAML/localStorage into a new persistence layer — because the v1.5 simulation already settled the persistence boundary (XDG YAML for backend settings, localStorage for reader chrome).
- **NOT** adding multi-named theme support — because v1.5 explicitly cut this; **one Custom slot per device** is the durable decision.
- **NOT** adding new tag entry shapes (autocomplete dropdown, suggestions) — because that's beyond the reported scope; only the visual affordance for "add tag" changes.
- **NOT** making search instant on Enter / dropping live search — because the user wants to keep search-as-you-type; only the debounce timing and first-key reliability change.
- **NOT** revisiting the reader header summarize button decision (spec D12) — because that decision stands; the issue is the missing overview-page wiring, not the header.

## Solution Direction

Eight fix areas, each scoped to a small surface. Where the v1.5 spec already named the desired behavior, this doc points to it rather than re-specifying.

### F1 — LLM Provider settings: collapse to a single CLI command field

- The settings page exposes only a `CLI command` field (string). The previous `Config Directory` field is **removed**. Rationale: the Claude / Codex CLIs resolve their own config via env / args; carrying a separate UI field added complexity without a use case the user actually wanted.
- Saving a non-default value (e.g. `claude-personal`) **persists durably** and **survives page reload and process restart**, with the typed value still visible in the field.
- The save action surfaces a visible success or error state; silent failure is not acceptable.

### F2 — Summarize button on Book Overview action area

- `BookOverviewView` shows the v1.5 spec FR-D3.2 button next to the Read action with three states:
  1. `Summarize N of M pending` (with spinner while a job is running)
  2. `K failed · Retry failed` (warning style, when failures exist and no work is pending)
  3. `✓ All N summarized` (status badge, non-interactive, when complete)
- Reader header has **no** such button (per spec D12 — that decision stands).
- The button reflects the same live job state the rest of the UI already observes; the user never sees stale counts.

### F3 — Add Tag affordance: text button → inline input on click

- Every site that currently shows a persistent "+ tag…" textbox switches to a small text button labeled `+ Add tag` (no border, no background fill).
- Clicking the button swaps it for a focused **inline input that is also borderless / underline-only** — the expanded state shouldn't visually jolt the user back into the very textbox-looking element they complained about.
- Pressing **Enter** or **blurring with content** commits the tag and reverts to the button. Pressing **Escape** or **blurring empty** reverts without saving.
- Sites in scope (the audit must cover all of them, not just the two known instances): book overview tag row, section tag row in the reader header, and any future site that mounts the same component.
- The single `TagChipInput` component changes once and the change propagates everywhere — but the requirement is that the **end UX** is consistent, not that the implementation share a component.

### F4 — Reader-settings state simplification

The user-visible reader-preset model collapses from **three sources of truth → one**:

- The **"Your Presets"** card group is **removed**. The user can no longer save named presets server-side; user-created server-side presets cease to exist as a concept.
- There is exactly **one Custom slot per device**, browser-local. Its values survive reload but do not sync across devices (consistent with the v1.5 decision on reader chrome state).
- The Custom slot has **no name field**. The card is simply labeled `Custom` — no ellipsis, no editor-opening connotation. It behaves like any other preset card: click to apply. The color picker / customizer is reached via an explicit edit affordance on the active Custom card, not by the card's own click. The earlier sub-idea of a placeholder-name + cancel-icon save flow (original Issue #4) is **dropped** in favor of this simpler model.
- "Currently applied" is a single piece of state with two possible kinds: a **system preset** (identified by ID) or **Custom**. The user always sees exactly one active indicator across the picker.
- **Clicking any system preset always applies it**, regardless of what was active before (including a Custom theme being edited). The active indicator moves to the system card within one frame; any partial customization the user was making is dropped without prompt.
- Re-fetching the preset list from the backend (e.g. on navigation) must not lose the active selection — the active selection survives across list refreshes.

This is the only structural change in the doc. Everything else is surface polish.

### F5 — Search command palette: debounce + first-key reliability

- Search-as-you-type **debounces at 300 ms** (up from 200 ms). Rapid typing produces exactly one trailing search request after the user pauses.
- The **first keystroke after opening the palette** is captured 100% of the time. No "the first letter sometimes vanishes" experience.
- While the debounce is in-flight, show a subtle indicator (the existing spinner is fine) so the user sees "I'm working on it".

### F6 — Cross-cutting: every fix surfaces a visible save / error state

The connecting theme behind several of these bugs is **silent failure** — the LLM settings save does nothing visible, the system preset click does nothing visible, the wrong tag button shape signals nothing. Every interactive surface in this doc must show its outcome:

- Save → success toast or inline confirmation.
- Save error → inline error with what to try.
- Click that mutates state → visible state change within one frame (active checkmark, applied colors, etc.).

This isn't a separate feature; it's a constraint that flows through F1–F5.

## User Journeys

### Journey 1 — Custom CLI binary

1. User has installed Claude CLI as `claude-personal` (separate auth profile).
2. Navigates to Settings → LLM Provider.
3. Edits the `CLI command` field to `claude-personal` and clicks **Save**.
4. **Sees** "Settings saved" confirmation.
5. **Reloads the page**. Field still reads `claude-personal`.
6. Triggers a summarize job; backend invokes `claude-personal` and the job runs.

### Journey 2 — Resume summarization from the book overview

1. User has a partly-summarized book (10 of 17 sections done, 2 failed).
2. Opens `/books/:id` (overview page).
3. **Sees** a `K failed · Retry failed` warning button next to **Read**.
4. Clicks it. Job starts; button changes to `Summarize 7 of 17 pending` with spinner.
5. Job completes. Button becomes `✓ All 17 summarized`.
6. Switches to the reader. Header has no summarize button — confirms spec D12.

### Journey 3 — Tagging a section

1. User is reading a section and wants to tag it `psychology`.
2. In the section tag row, **sees** a `+ Add tag` text button.
3. Clicks it. Button becomes a focused input.
4. Types `psychology`, presses **Enter**. Tag saves; chip appears; button returns.
5. Clicks `+ Add tag` again, presses **Esc**. Input dismissed; no tag created; button returns.

### Journey 4 — Custom theme then back to system

1. User opens the reader, opens Settings popover, picks the **Custom** card.
2. Adjusts background / accent / size, clicks **Save & Apply**.
3. Reading area reflects the custom colors. Custom card shows the active checkmark.
4. User clicks the **Sepia** system preset.
5. Custom is no longer active; Sepia is active and visibly applied. Reload preserves Sepia as active.
6. User clicks **Custom** again. The previously saved custom palette is restored (the slot remembers its values).

### Journey 5 — Quick search

1. User presses ⌘K. Command palette opens.
2. Types `motivation` quickly. **First keystroke is captured**; the search box reads `motivation`.
3. Stops typing. After ~300 ms, results appear.
4. Backspaces three characters. Spinner appears briefly; results refresh once after the typing pause.

### Error / Edge Cases

| Scenario | Trigger | Expected behavior |
|----------|---------|-------------------|
| LLM settings save with empty CLI command | User clears the field and saves | Inline error: "CLI command is required" or accept and fall back to auto-detect (whichever the v1.5 spec already settled). No silent reset. |
| User had a server-side custom preset before the migration | Existing `is_system=false` row in DB | Migration drops the row. User loses that preset; if they had matching values active, Custom slot is empty until they re-pick. Acceptable — this is a personal tool with one known user. |
| Active preset deleted (system list shrinks) | Future system preset list change | If the active preset ID disappears from the list, fall back to the first available system preset and clear `activePreset` cleanly. |
| Search palette reopened with prior query | User closes ⌘K, reopens | Either start clean OR remember the last query — current spec is silent; pick clean for predictability. |
| Tag input blurred with whitespace-only input | User clicks "+ Add tag", types spaces, blurs | Treat as empty; revert to button without saving. |

## Design Decisions

| # | Decision | Options Considered | Rationale |
|---|----------|-------------------|-----------|
| D1 | LLM Provider UI: single `CLI command` field; remove `Config Directory` | (a) keep both, (b) collapse to one, (c) put Config under "Advanced" | (b). The user has no concrete use case for a separate config dir; the CLI's own env/args cover it. Simpler UI, narrower bug surface. |
| D2 | Summarize button placement: `BookOverviewView` action area only | (a) overview only (per spec D12), (b) revert to reader header, (c) both | (a). The v1.5 spec already settled this; the issue is missing wiring, not a need to revisit placement. |
| D3 | Server-persisted user presets are removed | (a) delete + drop persistence, (b) migrate last-used into Custom slot, (c) hide UI but keep the table dormant | (a). One known user, pre-release; clean break is cheaper than migration logic and dormant tables rot. |
| D4 | One Custom slot per device, **no name** field | (a) name + placeholder + cancel icon (original suggestion), (b) name optional, (c) no name at all | (c). With user-created server presets gone, naming a single browser-local slot is friction without payoff. The slot is just "Custom". |
| D5 | "Currently applied" is one of: a specific system preset, or "Custom" — and survives list re-fetches | (a) reset on every refresh, (b) cache last selection in memory only, (c) preserve across refreshes by stable identity | (c). The user expects "what I picked stays picked" through navigation and refreshes. Stable-identity tracking is the contract; the implementation belongs to /spec. |
| D6 | System-preset click must clear any in-flight Custom-edit state | (a) always clear, (b) prompt the user, (c) merge | (a). The user clicked a system preset — that's an unambiguous "apply this" intent. Don't ask, don't merge. |
| D7 | Add Tag affordance is a text button that swaps to an inline input | (a) text button (Linear/Notion-like), (b) icon-only `+`, (c) keep input but strip border | (a). Most discoverable for a low-frequency action; matches well-known patterns. |
| D8 | Search debounce = 300 ms | (a) 250 ms, (b) 300 ms, (c) 400 ms, (d) Enter-to-search only | (b). 300 ms is the established sweet spot for type-ahead; backend is local so cost isn't a constraint, but UI flicker is. |
| D9 | First-keystroke reliability is a hard contract, not a heuristic | (a) longer debounce papers it over, (b) accept occasional drops, (c) require 100% capture | (c). 19/20 isn't acceptable for a search box; the spec author owns the implementation. |

## Acceptance Criteria

- [ ] Setting `CLI command` to `claude-personal` and clicking Save persists, surfaces a confirmation, and the value remains after reload.
- [ ] `Config Directory` field no longer appears anywhere in the LLM Provider settings UI.
- [ ] `BookOverviewView` shows the FR-D3.2 button with all three states; clicking the actionable states triggers a job and the button updates live.
- [ ] Reader header has no summarize button.
- [ ] Every "+ Add tag" entry point is a text button that expands to an input on click; pressing Esc / blurring empty reverts without saving.
- [ ] Reader settings popover has no "Your Presets" card group; the only non-system preset is the single Custom slot, with no name field.
- [ ] After defining a Custom theme, clicking any system preset switches the applied state to that system preset within one frame, with the active indicator on the system card and not on Custom.
- [ ] Reloading the page preserves the last-applied selection — whether it was a system preset or Custom.
- [ ] Existing user-created server-side presets are gone from the UI after upgrade; no orphan rows surface in any picker, list, or API response that the frontend reads.
- [ ] First keystroke after opening the ⌘K palette is captured 100% of the time across 20 consecutive opens (manual repro).
- [ ] Search debounce is 300 ms; rapid typing produces exactly one trailing request after the user pauses.
- [ ] Manual repro: every save / apply / commit across F1–F5 produces a visible confirmation, applied-state change, or explicit error within one frame — no silent no-ops.
- [ ] No regressions in v1.5's other reader features (highlights, annotations, tag CRUD, model selector, book overview cover).
