# V1.4 Pre-Ship Quality Pass — Requirements

**Date:** 2026-04-22
**Status:** Draft
**Tier:** 3 — Feature (bundled quality-pass scope)

## Problem

A full Playwright-driven audit of the packaged web UI (running `bookcompanion serve` on `localhost:8000`) surfaced 8 blocker-class defects, 7 significant bugs, and ~20 UX polish gaps — documented in [`docs/audits/2026-04-22_frontend_ux_audit.md`](../audits/2026-04-22_frontend_ux_audit.md). Two areas are currently unshippable:

1. **Summarization cannot be triggered from the upload wizard.** Three of four preset cards send IDs that don't exist on the backend; the one valid preset that goes through fails silently on most chapter-length sections with an empty-stderr `Claude CLI failed (rc=1)` and no UI indication. A user uploading their first book has no reliable path to a summarized book, and no visible signal that anything went wrong.
2. **Deep-linking into a section of the reader is broken.** Reloading `/books/:id/sections/:sid` or sharing the URL renders section 1, not the requested section — breaking both resume-on-refresh and shareable links.

Around those, smaller defects compound the "unfinished" feel: `/books/:id/summary` is a literal placeholder, chat responses render markdown as literal asterisks, the document title is permanently "Vite App", the global Annotations page lists highlights with no book or section context, and styling inconsistencies (casing, dates, labels) pepper every page.

### Who experiences this?

A personal user (the developer or a friend they've sent the wheel to) opening the packaged tool for the first time after install, uploading their first book, and trying to complete the full happy-path: *upload → summarize → read → highlight → chat → search*. Every one of those steps currently has at least one blocker or jarring UX break.

### Why now?

Two adjacent quality passes were scoped before this audit: [`2026-04-12-post-install-runtime-quality`](./2026-04-12-post-install-runtime-quality.md) (SQLite concurrency, SPA-fallback 404s, summarizer transactions) and [`2026-04-18-book-reader-ux-polish`](./2026-04-18-book-reader-ux-polish.md) (Copyright-landing, summary-tab fallthrough, image placeholders, content sanitization, book-detail progress UI, section-level Summarize CTA). The post-install doc has no feature commits yet — it's unimplemented. The reader-polish doc merged (`f7ce2ad`) but the end-to-end audit still reproduces multiple issues within its scope: empty-string summary markdowns stored as `has_summary: true`, null `default_summary_id` on sections with summaries, EPUB `.xhtml#filepos` heading-anchors surviving content sanitization, and the continue-reading banner parking on Copyright. v1.4 owns closure — the goals below subsume the unshipped / residual work, with explicit notes where a goal extends prior scope.

The 2026-04-22 audit is the first end-to-end, every-page run-through of the packaged SPA. Bundling all remaining fixes into a single v1.4 pass — while the same subsystems are already hot — avoids three separate rework windows and gets the tool to a state where "install the wheel and hand it to someone" is a credible demo.

## Competing Priorities (named tensions)

Two tensions run through this doc; calling them out now so design decisions later don't re-litigate them:

- **Ship fast vs. fix root causes.** Several goals (A4, A6, C4) address symptoms created by upstream backend silence — empty summaries stored, null defaults, missing stderr. The preference here is **fix the root cause even when a UI patch is cheaper**, because patching a symptom leaves the next user hitting the same bug. Named exceptions where we consciously take the cheap fix: **D2** (chat thread title — heuristic over LLM call; the status quo is ugly, not wrong) and **D6** (hybrid date format over full `Intl` i18n — good enough for a single-locale personal tool).
- **Closure vs. scope creep.** This doc deliberately bundles fixes across summarization, reader, chat, annotations, and global polish. Any work that grows beyond "observable audit finding closed" is an argument for deferring to v1.5 — see the non-goals list for what was consciously cut.

## Goals & Non-Goals

### Goals

**TL;DR** — v1.4 makes the packaged UI shippable by fixing (1) the upload → summarize path (preset wiring, per-section failure capture, one-click retry), (2) section deep-linking + the `/books/:id/summary` placeholder, and (3) ~20 polish items spanning chat markdown, annotation context, titles, casing, and dates.

Goals are grouped into six thematic clusters. Every goal has an observable acceptance criterion. Where a goal **extends** unshipped or partially-shipped work from a prior doc, that's called out at the end of the goal — but v1.4 owns the acceptance criterion, not the other doc.

#### Cluster A — Summarization trigger reliability

- **A1. Upload wizard presets map to real backend presets.** Acceptance: every preset card in the upload wizard, when selected and confirmed, returns `200` from the summarize endpoint for a valid book. No `Preset "X" not found` 400 is reachable by clicking a card.
- **A2. User-facing preset labels and descriptions are sourced from the backend, not hardcoded.** Acceptance: adding a new YAML preset file on disk makes it appear in the upload wizard after a server restart, with no frontend code change.
- **A3. Upload → Start Processing failures are visible in the UI.** Acceptance: if the summarize POST returns non-2xx, the wizard shows an inline error with the server's message and the Start Processing button returns to its non-busy state. Console-only errors are not a valid UX.
- **A4. Summarization failures are captured and surfaced per section.** Acceptance: when the LLM subprocess returns non-zero, a truncated error summary (type code + first ~500 chars of stderr) is stored alongside the section and displayed in a failure state on that section's Summary tab, plus aggregated at the book level ("3 of 27 sections failed · Retry failed"). Full stderr is additionally persisted to an append-only debug log file (path TBD in spec) for root-cause diagnosis; the UI only ever shows the truncated form.
- **A5. Per-section and per-book retry is one click.** Acceptance: from a failed section, a Retry button opens the preset picker with the **originally-chosen preset pre-selected** (see D11) and re-runs summarization for that section only. From the book detail page, a Retry failed sections button does the same but applies across the failed subset. Per-section preset used for a run is persisted so the pre-selection is deterministic.
- **A6. Empty-content summaries never count as "has summary".** Acceptance: if the LLM returns an empty or whitespace-only `summary_md`, the section is marked failed, not summarized; no Summary tab ever shows a blank page. Root-cause fix, not symptom — the summarizer service is the enforcement point, not the API layer.
- **A7. SQLite concurrency no longer 503s user reads during summarization.** Acceptance: with a summarization job running against a multi-section book, `PUT /api/v1/reading-state` returns 200 continuously; scrolling through a book during active summarization produces zero 5xx in the browser console for at least 60 seconds of continuous interaction. Mechanism (WAL tuning, busy_timeout, and breaking the summarizer's long transaction into per-section commits) is a spec decision. Extends [`2026-04-12-post-install-runtime-quality`](./2026-04-12-post-install-runtime-quality.md) which scoped but did not ship this.

#### Cluster B — Reader navigation and deep-linking

- **B1. Direct navigation to `/books/:id/sections/:sectionId` loads the requested section on hard-load, reload, and when pasted into a fresh tab.** Acceptance: visiting the URL from a cold tab renders the article for that section, updates breadcrumb/TOC state to match, and the correct tab pill is active. Reload preserves the same state.
- **B2. Section changes via TOC or arrow navigation update the URL (already works) AND reloading at that URL returns the user to the same section.** Acceptance: round-trip (navigate → copy URL → reload) lands on the same section.
- **B3. First open (no saved position) lands on the first summarizable section, not Copyright.** Acceptance: opening a book with no saved reading-state renders the first `is_summarizable: true` section, not front-matter. Extends [`2026-04-18-book-reader-ux-polish`](./2026-04-18-book-reader-ux-polish.md) — shipped but needs verification against the Gutenberg EPUB; the continue-banner variant is covered by E5.

#### Cluster C — Summary rendering and book-level surface

- **C1. `/books/:id/summary` ships a complete book-summary surface including generation.** Acceptance: the page shows (a) the book-level summary markdown when one exists, (b) per-section summary progress ("N of M summarizable sections summarized"), (c) a list of section summaries with titles and expand/collapse, (d) a **Generate Book Summary** button that opens a preset picker modal (same cards as upload) with the **last-used section summarization preset pre-selected** — see D11 — then queues the book-summary job, wired to the existing summarization pipeline with progress and error states. If generation fails mid-way, the button re-enables with a failure banner and a Retry CTA; partial state is preserved. When no book summary exists, the Generate button is primary and the page explains what a book summary is.
- **C2. Summary tab reliably renders the summary markdown when a summary exists with non-empty content.** Acceptance: for any section where `has_summary: true` and `default_summary.summary_md` is non-empty, the Summary tab renders that markdown. If `has_summary: true` but `summary_md` is empty, the section is treated as failed (per A6) and shows the failure state, not a blank Summary tab. Extends [`2026-04-18-book-reader-ux-polish`](./2026-04-18-book-reader-ux-polish.md) — the shipped version still has edge-case fallthrough to Original.
- **C3. Book detail page shows summarization progress and a Summarize pending sections button, verified against the Gutenberg *Art of War* upload.** Acceptance: after uploading the test EPUB, `/books/:id` (not the reader view) shows "N of M summarized" and a Summarize pending sections button that queues only missing sections. Extends [`2026-04-18-book-reader-ux-polish`](./2026-04-18-book-reader-ux-polish.md) — shipped components are in the codebase but not reached via the default route; verification gap.
- **C4. When a section has summaries but no default set, the most recent non-empty summary is auto-selected as the default.** Acceptance: opening a Summary tab for a section with `default_summary_id IS NULL` but summaries present renders the latest non-empty summary; `default_summary_id` is persisted so subsequent opens are consistent. One-shot migration + post-create hook; fixes the Porter book's current null-default state.

#### Cluster D — Chat, annotations, search polish

- **D1. Chat messages render markdown (bold, headings, bullets, code) — not literal characters.** Acceptance: a response containing `**Five Forces**`, numbered lists, and inline `code` renders with bold text, a real ordered list, and monospace — same rendering pipeline the reader article uses.
- **D2. AI chat threads auto-derive a title from the first user message.** Acceptance: the thread list shows a short summary (first ~40 chars of the first user message, trimmed on a word boundary) instead of "New Thread" for every thread. Heuristic, not LLM-generated (D-decision-table entry explains why).
- **D3. Chat shows the scope it has context for (conditional on backend capability).** Acceptance: the chat panel displays a scope badge. The affordance depends on what the spec phase discovers in the backend (see OQ1): if the backend currently supports only book-level context, the badge is a static display ("Book: {title}"); if section-scoped RAG already exists, the badge becomes a selector (Section ↔ Book). v1.4 ships whichever matches the current backend — no new backend capability is promised here.
- **D4. `/annotations` page shows book title + section title + "Go to source" link on every annotation row.** Acceptance: clicking a highlight row navigates to that section with the highlight scrolled into view (scroll behavior is a stretch; nav to the section is the must-have).
- **D5. Command palette shows a loading state while debouncing and a "No results" state when the query returns zero matches.** Acceptance: typing and pausing shows a spinner (or skeleton rows) until results arrive; if results are empty, shows "No results for '{query}'" plus a link to the full search page.

#### Cluster E — Global polish

- **E1. Per-route document title.** Acceptance: opening any page updates `document.title` to "{Page} — Book Companion" (library, book title for reader, etc.). No tab shows "Vite App".
- **E2. Naming consistency pass.** Acceptance: (a) left-nav label matches page `<h1>` and sidebar-tab label for Annotations/Notes, Concepts, Library, Settings (see D5). (b) All user-visible filter option labels use **Capital-Case-With-Spaces**; acronyms stay uppercase. Specifically: status filter renders "All Statuses", "Parsed", "Summarizing", "Completed"; format filter renders "All Formats", "EPUB", "PDF", "MOBI". Internal API enums remain lowercase — only the display layer transforms. (c) Reader page `<h1>` reflects the current book title, not the literal word "Reader".
- **E3. Human-readable, locale-aware dates.** Acceptance: no date renders as ambiguous DD/MM or MM/DD in the UI. Rendering rule follows D6 (hybrid relative/absolute). Consistent across library table, annotations list, summary metadata, and chat timestamps.
- **E4. Button casing matches convention.** Acceptance: "Export Markdown" (not "Export MARKDOWN"); no other all-caps button labels.
- **E5. Continue-reading banner never parks on non-summarizable front-matter.** Acceptance: the reading-state tracker skips persisting sections whose `section_type` is in the front-matter set (reuse the set from 2026-04-18 — `copyright`, `acknowledgments`, `dedication`, `title_page`, `table_of_contents`, `colophon`, `part_header`) as resume points. If only front-matter sections have been visited, the banner is hidden rather than surfacing a meaningless resume.
- **E6. TOC "has summary" indicator is labeled.** Acceptance: the per-section summarized-badge uses "✓" plus `aria-label="Summary available"` and a `title` tooltip. No bare letter "S" with no context.
- **E7. Image alt text is meaningful.** Acceptance: images inside rendered content have `alt` derived from nearby caption or figure title where available; decorative images (height ≤ 2px or explicitly flagged) carry `alt=""` (empty) so screen readers skip them rather than saying the literal word "image".
- **E8. Structure step in upload wizard uses friendly type labels.** Acceptance: `table_of_contents` renders as "Table of Contents", `introduction` as "Introduction", etc. No underscores or lowercase enum values shown to the user.
- **E9. Upload confirmation uses the user-facing preset label, not the internal ID.** Acceptance: after Start Processing, confirmation copy reads "…with the Practitioner preset" (or whichever label the user picked), not "…with the practitioner_bullets preset".
- **E10. Stored highlights are visually distinct from the browser's live text selection.** Acceptance: the saved-highlight color (in reader content) differs noticeably from the default browser-selection color so the user can tell "saved" from "currently selecting" at a glance. CSS-only change.

#### Cluster F — Defect cleanups and classifier quality

- **F1. Reader Settings pane opens without a runtime error.** Acceptance: opening ⚙ on any section produces zero console errors. The underlying `e.presets.filter is not a function` TypeError no longer occurs; the presets dropdown (if any) populates correctly.
- **F2. In-content EPUB anchors are neutralized across all patterns observed in the audit.** Acceptance: `href="#filepos..."`, `href="CR!…xhtml#..."`, and heading self-anchors all render as plain text. External `http(s)://` links stay clickable with `target="_blank" rel="noopener noreferrer"`. Extends [`2026-04-18-book-reader-ux-polish`](./2026-04-18-book-reader-ux-polish.md) — link-policy shipped but doesn't catch the `.xhtml#filepos` pattern observed on Porter headings.
- **F3. Metadata step in the upload wizard is removed from the step indicator** (decided — see D8; editable metadata is an explicit non-goal). Acceptance: the wizard step indicator shows Upload → Structure → Preset → Processing (four steps, not five); no step is silently skipped.
- **F4. Structure classifier correctly tags bylines, footnotes, licenses, content-TOC sections, and author-name-only sections.** Acceptance on the Gutenberg *Art of War* EPUB: (a) sections like "LIONEL GILES, M.A." classify as `title_page` (not `chapter`); (b) "Footnotes" classifies as `notes`; (c) "THE FULL PROJECT GUTENBERG™ LICENSE" classifies as `license` or `appendix`; (d) byline/author-name-only sections are excluded from default summarization runs (not summarized unless the user explicitly overrides per-section). Extends 2026-04-18 classifier work to cover the Gutenberg edge cases the audit exposed.

### Non-Goals (explicit scope cuts)

- **NOT building a new ingestion pipeline or parser rewrite**, because the classifier fixes in F4 are extensions of the existing title-regex patterns; a full rewrite would eclipse ship-readiness.
- **NOT adding new summarization presets or editing prompts**, because the bug is that the frontend doesn't know what presets exist — not that the preset library is wrong. The five existing YAML files are treated as source-of-truth.
- **NOT redesigning the upload wizard flow**, because the flow works once its preset-ID bug is fixed; a redesign would trade a shippable fix for scope.
- **NOT building an editable metadata step in the upload wizard**, because dropping the skipped step (F3) resolves today's confusion; editable metadata is a separate feature for v1.5+.
- **NOT reworking the AI chat UX beyond markdown rendering, auto-title, and scope indication**, because threaded conversation, message editing, citation UI, and context controls are a proper feature pass — v1.4 covers the three things that make today's chat look broken.
- **NOT surfacing the `database_migrations_behind` warning in the UI as a banner**, because the server still runs with an out-of-date schema; autoruning migrations on startup (or a CLI `init` check with a clear error) is a better solve. Deferred rather than a v1.4 goal.
- **NOT adding localization/i18n**, because E3 (date format) can be solved with a single locale-aware helper without introducing an i18n framework; full translation is out of scope.
- **NOT re-architecting error reporting into a toast/notification system**, because inline errors (A3) and per-section banners (A4) are enough for v1.4. A global notification system can come later.
- **NOT fixing decorative-divider image rendering oddities (217×1 images with padding)**, because they render correctly at their natural dimensions; only the alt-text issue (E7) matters.

## User Experience Analysis

### Motivation

- **Job to be done:** Install the tool, upload a non-fiction book, get usable summaries per chapter plus a book-level view, and then read/highlight/chat with confidence that the tool is finished enough to depend on.
- **Importance/urgency:** High for Maneesh personally (daily use is the validation loop for this project) and for anyone he hands the wheel to. A user who hits the upload-wizard preset bug on their first book has no reason to come back.
- **Alternatives:** The CLI (`bookcompanion summarize <id> --preset practitioner_bullets`) works today; a user who knows the CLI exists can route around the UI. But the UI is the front door; if it looks unfinished, the CLI is never discovered.

### Friction Points

| Friction Point | Cause | Mitigation |
|---------------|-------|------------|
| "I clicked Start Processing and nothing happened." | Wizard POST returns 400, no inline error | A1 + A2 + A3 — presets work, and when anything else fails, the user sees it inline. |
| "Some chapters have summaries, some don't, and I can't tell why." | Silent Claude-CLI failures store no error context | A4 + A5 — failure reason stored and shown; one-click retry. |
| "I bookmarked a chapter and when I clicked the link, it went to the cover." | `/books/:id/sections/:sid` deep-link broken on hard-load | B1 — router resolves sectionId from the URL on mount. |
| "The Summary tab looks selected but there's no summary on it." | Empty-string summaries marked `has_summary: true` | A6 + C2 + C4 — empty summaries treated as failed; defaults auto-selected. |
| "I clicked the Book Summary link and got a blank page with the text 'BookSummaryView'." | Route is an unimplemented stub | C1 — ship a real view with a Generate button. |
| "The chat answered me in Markdown but it looks like code." | Chat doesn't render markdown | D1 — wire markdown-it + DOMPurify into the chat bubble. |
| "My highlight doesn't tell me which book it's from." | `/annotations` has no source metadata | D4 — add book title, section title, and a link to source. |
| "Every browser tab says 'Vite App' so I can't find the right window." | Title never updated per-route | E1 — per-route `document.title` via router guard. |
| "I can't tell what I just highlighted from what I'm currently selecting." | Stored highlights use ~same color as browser selection | E10 — distinct saved-highlight CSS color. |

### Satisfaction Signals

- Upload → summarize → read completes in one session with no console errors and no silent failures.
- Shareable/bookmarkable URLs for sections and book summaries work.
- Every annotation leads back to its source in one click.
- No page is an obvious placeholder.
- Naming, casing, and dates are internally consistent so the UI doesn't feel "stitched together by multiple authors."

## Solution Direction

Six parallel workstreams corresponding to the goal clusters. They can ship independently behind a single release cut; there is no forced ordering beyond A7 (db concurrency) being a prerequisite for reliable A4-A6 testing.

### 1. Summarization reliability (Cluster A)

**Backend.** Add a `GET /api/v1/summarize/presets` endpoint that lists the YAML files under `app/services/summarizer/prompts/presets/`, returning `{id, label, description, facets}` for each. Register this route **before** the SPA fallback so it returns 404 when missing (same pattern as the post-install-runtime-quality doc scoped). In `ClaudeCodeCLIProvider` / `CodexCLIProvider`: capture subprocess stderr, persist a truncated + typed failure record to the database on each section, and append the full stderr to a rotating debug log file. Treat empty-or-whitespace summary markdown as a failed job, not a success, at the summarizer-service layer.

Close A7 in the same workstream (the mechanism — WAL tuning, `busy_timeout`, per-section commit boundaries — is a spec decision) since it's a prerequisite for observing A4 reliably: reading-state writes must not 503 while the summarizer is under test.

**Frontend — upload wizard.** Replace the hardcoded preset list in `PresetPicker.vue` with a fetch of the new endpoint on mount. On POST failure, render the server's error message inline and clear the busy state. Confirmation copy uses the chosen preset's user-facing label.

**Frontend — reader and book detail.** Where a summary failed, the Summary tab shows the stored error (truncated, expandable) plus a Retry CTA. The book detail page tallies successes / failures and exposes Retry failed sections.

### 2. Reader deep-linking (Cluster B)

The router currently reads `bookId` on mount but `sectionId` is resolved lazily from the reader store, which isn't populated on cold navigation. The fix is a route resolver / `beforeEnter` guard that fetches the section before the component renders — or a `watch` on the route param that triggers the same section-load the TOC click uses. Exact shape is a spec decision. Verify B3 landing behavior against the Gutenberg EPUB while touching the router logic.

### 3. Book-level summary surface (Cluster C)

Replace `BookSummaryView.vue` stub with a real implementation: book-summary markdown (rendered via the existing markdown pipeline), progress counter, section summary list (accordion), **Generate Book Summary** button wired to the summarization job pipeline with SSE progress/error states. When no book summary exists, the Generate button is the primary CTA.

Auto-select the default summary on fetch: if `default_summary_id IS NULL` but summaries exist, pick the most recent non-empty one and persist. This is a one-shot migration + a post-create hook. Fold into the same migration that closes the 2026-04-18 residual (empty-summary cleanup).

### 4. Chat / annotations / search polish (Cluster D)

- Wire the existing markdown-it + DOMPurify pipeline (already used for reader content) into the chat bubble component.
- On first user message, derive a thread title (simple heuristic: first ~40 chars, trim on word boundary).
- Chat panel shows scope badge: "Section: Introduction" / "Book: Understanding Michael Porter" / "Whole library". Actual scope capability is a spec read — if backend currently only supports whole-book, the badge is static; if it supports section-scoped already, it becomes a selector.
- Extend annotation API response with `book_title` and `section_title`; render them on `/annotations` rows with a link to `/books/:id/sections/:sid`.
- Add loading + empty states to CommandPalette.

### 5. Global polish (Cluster E)

- Router `afterEach` hook updates `document.title` from route meta.
- Naming audit: rename left-nav "Notes" → "Annotations" (decided — D5). Reader `<h1>` becomes book title. Filter options normalized.
- Date helper: `formatDate(iso, style = 'relative' | 'long')` defaulting to hybrid (relative < 7 days, absolute otherwise).
- Remove all-caps button labels.
- `ReadingStateService` skips persisting sections whose `section_type` is in the front-matter set (reuse the set from 2026-04-18).
- TOC summary-badge: replace "S" with "✓" + `aria-label="Summary available"` + `title` tooltip.
- Markdown renderer: when emitting `<img>`, if the image dimensions suggest it's decorative (height ≤ 2px) or there's no caption source, emit `alt=""`; otherwise use nearby caption text.
- Structure step: friendly-label helper for section types.
- Saved-highlight CSS: distinct color variable from `::selection` rules.

### 6. Defect cleanups and classifier (Cluster F)

- Track down `e.presets.filter is not a function` in `BookDetailView`: whatever store/prop is being filtered is sometimes an object or `null`; initialize or coerce.
- Drop the Metadata step from the upload step indicator.
- Classifier extensions for byline-only sections, Gutenberg license blocks, "Footnotes" as notes, and author-name-only sections (excluded from default summarization). Piggyback on the 2026-04-18 classifier patterns.

## User Journeys

### Primary Journey (Happy Path: upload → summarize → read → chat → share)

1. User navigates to `/upload` and drops a fresh EPUB.
2. Upload + parse completes; wizard advances to Structure review where sections are shown with human-readable type labels.
3. User picks a preset card; confirmation text reads "Your book is being summarized with the **Practitioner** preset."
4. View Progress lands on `/books/:id`; page shows progress counter "0 of 17 summarized" and a list of sections.
5. As sections summarize, the counter updates ("5 of 17 summarized"). TOC badges (✓) appear next to completed sections.
6. User clicks into a chapter. Summary tab on a completed section renders markdown. User switches section, reloads the tab — correct section still rendered.
7. User copies the URL, pastes into a new tab — loads the same section directly.
8. User opens the AI chat, asks a question, receives a markdown-formatted response. Thread list shows an auto-derived title.
9. User selects text → Highlight. Later they open `/annotations` and click the highlight → navigated back to the source section.
10. User navigates to `/books/:id/summary`, clicks Generate Book Summary, watches progress, reads the synthesized summary.
11. Document title on each tab reflects the current page — they can find the right window in a long tab bar.

### Error Journey 1 — A chapter failed to summarize

1. User opens a chapter. Summary tab shows **Summary failed**: "CLI exited with code 1: [truncated stderr]" plus a **Retry** button and a **View full log** expandable link.
2. User clicks Retry. Status changes to **Generating summary…**
3. On success: rendered summary replaces the failure state. On repeated failure: same failure state with updated timestamp and error message.

### Error Journey 2 — Upload-wizard preset fails

1. User picks a preset and clicks Start Processing.
2. POST returns non-2xx (rare after A1, but possible — e.g., LLM provider unavailable).
3. Inline error below the preset cards: "Could not start summarization: {server message}. Try another preset or check that Claude/Codex CLI is installed."
4. Start Processing button re-enables; user can retry or change preset.

### Error Journey 3 — Broken deep-link (404 section)

1. User pastes an out-of-date URL with a stale `sectionId` (book re-imported since).
2. Router resolves and fetches; API returns 404.
3. Reader shows "This section no longer exists" and offers a link to the book's TOC or first section.

### Empty States & Edge Cases

| Scenario | Condition | Expected Behavior |
|----------|-----------|-------------------|
| Command palette empty results | Query matches nothing | "No results for '{query}'" + link to `/search?q={query}` |
| Chat panel on front-matter section | Section has no summarizable content | Chat opens scoped to the book, not the section; scope badge reflects this |
| `/annotations` with no annotations | First-time user | Existing "No annotations yet" empty state — extend copy to mention how to create highlights |
| `/books/:id/summary` for unsummarized book | No book summary + no section summaries | Generate Book Summary button is the primary CTA; page explains what a book summary is |
| Chat sending while offline / backend down | Fetch fails | Inline error in the thread, message stays in the input for editing |
| Preset list fetch fails at wizard mount | Backend down at wizard load time | "Loading presets…" then, after timeout, "Could not load presets — the server may not be running. Retry." Never a silent empty picker. |
| Generate Book Summary clicked with no section summaries yet | User tries to generate book summary before any section is summarized | Button either disabled with tooltip "Summarize sections first" OR allowed but warns "Summary quality will be lower without section summaries first" — spec decision (OQ2) |
| Generate Book Summary fails mid-generation | LLM returns error partway through | Button re-enables with failure banner showing truncated error + Retry CTA; partial state (prior sections read) preserved |
| Generate / Retry preset picker with no presets loaded | Preset fetch failed at mount | Modal shows "Could not load presets" + Retry load button; submit disabled |

## Design Decisions

| # | Decision | Options Considered | Rationale |
|---|----------|-------------------|-----------|
| D1 | Presets are surfaced via a new `GET /summarize/presets` endpoint, not embedded in the SPA bundle. | (a) Fetch at runtime (chosen). (b) Bake into frontend build-time via Vite plugin reading YAML.
