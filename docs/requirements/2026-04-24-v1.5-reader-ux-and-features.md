# V1.5 Reader UX & Features — Requirements

**Date:** 2026-04-24
**Status:** Draft
**Tier:** 3 — Feature (bundled reader-UX + new-feature scope)

## Problem

A focused pass through the packaged reader, one book deep (*Understanding Michael Porter*), surfaced 14 issues that fall into three kinds of gap: real render/routing bugs, polish items that make the tool feel unfinished, and two net-new surfaces that are simply missing. The reader currently fails at four things a user should be able to take for granted — **images in summaries render** (the LLM preserves them but the backend never rewrites placeholders on summary markdown), **blockquotes look like quotes** (publisher nested-blockquote styling hacks survive parsing and double the left border), **next/previous navigation updates the URL** (the buttons mutate Pinia state but never call `router.push()`, so reload breaks and share links land on the wrong section), and **a book-level summary is reachable from the UI** (the `/books/:id/summary` page shipped in v1.4 but nothing links to it, and `/books/:id` still redirects into the reader).

Around those, smaller papercuts compound the "unfinished tool" feel: the reader has one fixed theme (no presets, no sepia, no night mode, no custom colors), highlights show only in the Annotations sidebar (never inline in the text), the Annotations tab mixes every chapter's notes into one flat list, adding a note opens a browser `window.prompt()`, the search box is janky because the request fires on every keystroke, Ask AI appears to do nothing on click, books without a cover show an empty tile, the "Summarize pending" button is an icon with a tooltip that disappears when the work is done, and two features (**tags** and **per-summarization model selection**) are either stubbed-but-dropped in schema or hardcoded and unreachable from the UI.

v1.5 bundles these into one pass because they share surfaces (reader, settings, book detail, annotations) and because leaving any of them in their current state makes the tool feel consistently rough rather than specifically broken.

### Who experiences this?

A personal user — the developer or someone they've handed the wheel to — who has moved past "can I install and upload a book?" (v1.4's scope) and is now trying to actually *read and extract value from* a non-fiction book. Every reported issue comes from a single real session reading the Porter book end-to-end: theming choices while reading in the evening, highlights that the user wanted to see inline in the text the next day, tags they tried to add and couldn't find, a book summary they knew existed somewhere but couldn't navigate to.

### Why now?

v1.4 made the tool shippable (upload → summarize → read without blockers). This pass is about making it pleasant to use for more than ten minutes. Three forcing functions:

1. **Theming absence is the single most-felt gap during extended reading.** Every comparable tool (Kindle, Apple Books, Readwise Reader, Instapaper) ships 3–6 theme presets out of the box. We ship one, and it's bright white.
2. **Tags are schema-stubbed-then-dropped in the DB migration** (`e152941ea209_initial_sqlite_schema.py`). Leaving this half-built signals "feature coming" to anyone reading the code; either build it or remove the stubs. v1.5 builds it.
3. **The bug shape is cheap to fix right now.** #1 (summary image rewrite), #12 (router.push), #13 (button text), #9 (search debounce) are all one-file fixes while the relevant code is hot. Deferring them means relearning the code later.

## Competing Priorities (named tensions)

- **Scope discipline vs. closing the session.** Tags is a full feature build inside what would otherwise be a polish pass. The preference is to ship the full tag system (book + section, AI-seeded at book level, manual edit) rather than stub it again — the current half-state is worse than either extreme. Named counter-exception: **section-level AI tag seeding is deferred** to v1.6 because per-section LLM cost multiplies and we have no evidence users want that density.
- **Custom color freedom vs. accessibility.** Custom hex input is a stated user ask but the dominant reading-app pattern (Kindle, Apple Books, Instapaper) keeps users on a curated palette because unconstrained custom colors produce unreadable themes. v1.5 resolves this by shipping a curated palette as the primary affordance and exposing "Custom…" as a secondary action that requires passing through a live WCAG contrast readout. Warn, don't block — the user owns the final call.
- **Device-global settings vs. per-book nuance.** Reader settings (theme, font, colors) are device-global in localStorage. A user reading fiction and non-fiction side-by-side might want different themes per book, but we have zero evidence for that use case and per-book overrides multiply surface area (invisible-to-user state divergence, export/import complexity). Explicitly a v1.6+ consideration.

## Goals & Non-Goals

### Goals

**TL;DR** — v1.5 makes the reader pleasant for extended sessions by shipping (1) six theme presets + custom color/text-color with WCAG-checked hex input, (2) inline highlights with per-theme color + show/hide toggle, (3) a right-side markdown note editor, (4) a scoped annotations tab with scroll-to-source, (5) a real `/books/:id` book overview page, (6) a full tags system (book + section, AI-seeded at book level), (7) LLM model selection from a config-file-driven dropdown, and (8) ~8 rendering/routing/polish fixes.

Goals are grouped into seven clusters. Every goal has an observable acceptance criterion. Numbering maps back to user report items in brackets.

#### Cluster A — Reader rendering bugs

- **A1. Images in section summaries render identically to original content. [#1]** Acceptance: for any section whose default summary markdown references images, the Summary tab renders them as working `<img>` tags resolving to `/api/v1/images/{id}` — never as literal `(image:ID)` or unresolved `__IMG_PLACEHOLDER__` text. Root-cause fix at the backend summarizer save path (`from_placeholder()` must be called on summary markdown before persistence), not a frontend patch. Includes a one-time migration that rewrites placeholders in already-stored summaries. **Migration edge case:** if a placeholder references an image_id no longer in the DB (deleted, re-imported), strip the placeholder entirely (empty string) and log a per-section warning — never leave broken syntax in the stored markdown and never fail the migration.
- **A2. Blockquotes render with a single left border, not nested wrappers. [#2]** Acceptance: on the Porter book section 9 (and the 7 other sections with nested `> >` markers in `content_md`), the "value proposition" pull-quote renders as one blockquote with one blue bar — not two stacked. Fix at parser time: collapse nested `<blockquote><blockquote>` to a single level during EPUB→markdown conversion, and strip HR markers (`---`) that appear as the first or last line inside a blockquote. One-time migration re-normalizes existing `content_md`.

#### Cluster B — Reader theming & settings

- **B1. Reader ships six named theme presets out of the box. [#3]** Acceptance: opening the reader settings popover shows six named presets (e.g., *Paper, Sepia, Slate, Solarized, Mocha, Night*) as selectable cards. Each preset bundles background color + text color + font + line-height into a coherent look. Selecting a preset applies immediately and every surface inside the reader (article, summary, highlights, toolbar, sidebar) inherits the theme.
- **B2. Background-color palette is 6–7 curated colors plus a Custom… action. [#3.1, #3.2]** Acceptance: below the presets, a color row shows 6–7 curated background colors (two light, two warm, two cool, one dark — exact palette a spec decision) as clickable swatches. Below that, a Custom… button opens a color wheel with a hex input field and a live contrast-ratio readout against the current text color. If the combination falls below WCAG AA (4.5:1), show a warning badge but allow the user to apply it anyway.
- **B3. Text-color palette mirrors the background-color affordance. [#3.3]** Acceptance: a second color row offers 6–7 curated text colors + Custom…, same WCAG feedback, independent of the selected background preset.
- **B4. Non-preset customizations require explicit Save & Apply to persist.** Acceptance: when any preset is active and the user changes any axis (bg color, text color, font, size, line-height), the reader **previews the change live** but does NOT yet persist it. The settings popover shows a prominent "Save & Apply as Custom" CTA above the preset row indicating an unsaved custom combination. If the user clicks Save & Apply, the combination is persisted as the single "Custom" slot and becomes the active state (all preset cards deselect; a "Custom" card/indicator takes their selected state). If the user dismisses the popover, reloads, or navigates away without saving, the change is discarded and the reader reverts to the last-saved state on next render. Re-selecting any preset card snaps all axes back to that preset's defaults and clears any pending-unsaved state. One Custom slot total — saving overwrites any prior Custom.
- **B5. Reader settings persist across sessions per device. [#3]** Acceptance: on Save & Apply (for custom combinations) OR on preset selection (presets auto-save since they're deterministic bundles), the active state is written to `localStorage` under a single top-level key. Closing the tab, reloading, or opening a different book on the same device applies the saved state (named preset OR the Custom slot, per B4). Clearing site data resets to the default preset. No DB writes; no cross-device sync in v1.5.

#### Cluster C — Annotations & highlights UX

- **C1. Adding a note from a selection opens a custom right-side markdown editor, not `window.prompt()`. [#4]** Acceptance: clicking "Note" on a text selection opens (or focuses) the sidebar in a Note composition state: the selected text is shown as a collapsible quoted block at the top, a markdown textarea fills the body with a live-preview toggle, a tags chip input accepts one or more tags, Save and Cancel buttons bottom-right. Saving persists the annotation; Cancel discards. The text selection is preserved across the open action.
- **C2. Long notes render with a "see full" collapse in the Annotations list. [#4]** Acceptance: in the Annotations tab, any note body exceeding ~200 characters renders truncated with an ellipsis and a "See full" inline link. Clicking expands the row in place. Short notes (< 200 chars) render fully, no toggle.
- **C3. Annotations tab defaults to the current section, with an expandable group for other sections. [#5]** Acceptance: opening the Annotations tab on any section shows only that section's annotations (highlights + notes), ordered by position in the text. A muted "Show N annotations from other chapters" row below the list expands to reveal all other-section annotations, grouped by section title with each group expandable. State of the toggle persists within the session (collapses back to current-section-only on section navigation).
- **C4. Clicking an annotation row scrolls to its highlighted text. [#5]** Acceptance: clicking any annotation row — whether in the current section or an expanded other-section group — navigates the reader to that section (route change) if needed, then smooth-scrolls the highlighted range into view centered in the viewport, with a ~1.5s background-color pulse on the target `<mark>` element so the user spots it.
- **C5. Highlights render as inline background marks in the article body. [#6]** Acceptance: every highlight stored for the current section renders as a `<mark>` (or equivalent wrapper) inside the rendered article content, with a theme-aware translucent yellow background that adapts across presets (full saturation on Paper, desaturated on Night). Applies to BOTH the Original and Summary tabs where the highlighted text appears. Overlapping highlights combine visually without data loss.
- **C6. Show/hide inline highlights toggle in Annotations tab. [#6]** Acceptance: the Annotations tab header shows a "Show inline" toggle (default: on). Turning it off removes `<mark>` styling from the article content without deleting the annotation data; turning it back on restores. Setting persists in localStorage (per B4).

#### Cluster D — Reader navigation & book surfaces

- **D1. `/books/:id` renders a real book overview page, not the reader. [#11]** Acceptance: visiting `/books/:id` directly (breadcrumb, library click, deep link) renders a dedicated overview view with: (a) header (cover or fallback, title, author, format, progress bar "N of M sections read, K of M summarized", Resume reading CTA going to last-visited section); (b) book summary zone (rendered markdown if one exists, Generate Book Summary CTA opening a preset picker if not, and a "View full summary" link to `/books/:id/summary` for focused reading); (c) TOC with per-section summary status (✓ summarized, ⚠ failed, — pending), per-section tag chips, and a "Summarize pending" action (see D3). The reader remains at `/books/:id/sections/:sid` only.
- **D2. Next/Previous section toolbar buttons update the URL. [#12]** Acceptance: clicking Next or Previous in the reader header (or using a random/jump action) triggers a `router.push` to `/books/:id/sections/:nextId`. Reloading the page returns the user to the same section. Copying and pasting the URL into a fresh tab lands on the same section. Matches the breadcrumb `<router-link>` behavior.
- **D3. "Summarize pending sections" button is a single source of truth on the book overview. [#13]** Acceptance: the button lives **only on the book overview page (D1)** — the reader header no longer shows it. On the overview, the button displays "Summarize N of M pending" when work remains (N > 0). On click: button disables, an inline spinner appears, and the text updates to "Summarizing {current} of {N}…" driven by the existing SSE `section_started` / `section_completed` events via the shared `summarizationJobStore`. When N = 0, the button remains visible in a muted success state reading "All M summarized" — not hidden. No tooltip. Reader users access the button by navigating back to the book overview (breadcrumb or new routing per D1/D2).

#### Cluster E — Tags system (greenfield)

- **E1. Re-enable Tag and Taggable schema with polymorphic associations to Book and BookSection.** Acceptance: the DB has active `tags` and `taggables` tables (re-added via a new Alembic migration that replaces the v1-initial drop) with unique tag names (case-insensitive), optional color, and a polymorphic join supporting `taggable_type IN ('book', 'section')`. `Book` and `BookSection` expose a `tags` relationship via eager-loaded query paths.
- **E2. Book-level tag CRUD via UI and API.** Acceptance: the book overview header shows book tags as chips with a + affordance to add or edit. Adding a new tag auto-completes against existing tags in the library. Tags can be removed per book. API endpoints exist for list / add / remove at book scope. Tag chips on library cards become filterable (click a chip on a library card → filters the library by that tag).
- **E3. Section-level tag CRUD in the reader header and overview TOC.** Acceptance: the reader header shows the current section's tags as small chips above the section title with a + to add/edit inline. The book overview TOC shows the same tags next to each section row, editable from the TOC. Same auto-complete behavior as E2. API endpoints at section scope.
- **E4. LLM suggests 3–5 book tags on FIRST book-level summary completion; user accepts/rejects.** Acceptance: the first successful book-level summarization job returns a `suggested_tags` list alongside the summary. The book overview header shows a "Suggested tags" pill group with each tag as a pending chip carrying check/X micro-affordances. User accepts individual tags into the book's applied tags; rejected suggestions do not persist. Regenerating the book summary later does NOT produce new suggestions — the AI-suggestion flow is one-shot per book. Subsequent tagging is manual (per E2). v1.5 caps section-level AI seeding as a non-goal (see below).

#### Cluster F — LLM configuration

- **F1. Settings page exposes a model selector with curated models loaded from a config file. [#14]** Acceptance: the Settings page has an LLM section showing the detected provider(s) (`claude`, `codex`) and a dropdown listing candidate models for the active provider. The candidate list is loaded from `backend/app/config/models.yaml` (shipped with the package) — updating models across releases does not require a code change. Changing the dropdown writes to `settings.llm.model` and applies to all subsequent summarization jobs.
- **F2. "Custom…" option in the dropdown exposes a free-text model name input.** Acceptance: the last dropdown option is Custom…; selecting it reveals a text field where the user types an arbitrary model alias (e.g., `claude-opus-5` before it's in the curated list). The value flows through the existing per-request `model` parameter in `ClaudeCodeCLIProvider.generate()` / `CodexCLIProvider.generate()` unchanged. If the provider subprocess fails (exit code non-zero with an error citing the model), the UI surfaces the error message from F3 below.
- **F3. Model selection errors surface visibly in the UI.** Acceptance: if a summarization job fails because the CLI returned a "model not found" or equivalent error, the book overview progress banner and the section-level Summary tab failure state both show the truncated stderr (same truncation policy as v1.4 A4). User can retry after changing the model in Settings.

#### Cluster H — CLI parity

The Typer CLI is the tool's second UI and must reach parity with the new surfaces v1.5 introduces. Code recon surfaced two dormant command files (`tags_cmd.py`, `annotations_cmd.py`) that exist but are **not registered in `main.py`** — they mirror the half-built tag schema. v1.5 wires them up and extends them.

- **H1. Tag CLI covers books + sections, with library filtering.** Acceptance: `bookcompanion` has a `tag` subcommand group registered in `main.py` with commands: `tag add <scope> <id> <name>` (scope ∈ {book, section}), `tag remove <scope> <id> <name>`, `tag list` (all tags in library with counts), `tag list --book <id>`, `tag list --section <id>`, `tag filter <name>` (lists books with that tag). Existing `tags_cmd.py` is extended to accept the scope argument (currently hardcoded to "book") and registered.
- **H2. CLI `tag suggest` reads stored suggestions from the book-summary run — it does NOT trigger new LLM calls.** Acceptance: `bookcompanion tag suggest <book_id>` prints the 3–5 AI-suggested tags stored with the book from its first book-level summarization. Output is a numbered list plus hint text: "Run `bookcompanion tag add book <id> <name>` for each you want to keep." If the book has no summary yet, print: "No suggestions yet — run `bookcompanion summarize <id> --book-summary` first." Same one-shot semantics as the UI: suggestions are generated exactly once (during the first successful book-summary job) and this CLI command only retrieves them. No on-demand generation, no Re-suggest path.
- **H3. Model CLI exposes the config-file-driven list.** Acceptance: `bookcompanion model list` prints the candidate model list loaded from `backend/app/config/models.yaml`, grouped by provider (`claude:`, `codex:`). `bookcompanion model current` prints the currently-selected `settings.llm.model` plus the detected provider. `bookcompanion model set <name>` is an alias for `config set llm.model <name>` and validates against the config file (warns but allows if the name isn't in the list — Custom…-equivalent).
- **H4. Summarize-pending parity in CLI.** Acceptance: `bookcompanion summarize <book_id> --only-pending` runs summarization only on sections with `default_summary_id IS NULL` AND `section_type ∈ SUMMARIZABLE_TYPES`. Existing `summarize` without the flag retains current behavior (all missing + optionally re-do). Output prints "Summarizing N of M pending sections" at start, matching the UI's D3 language.
- **H5. Annotation CLI registered as read-only list views.** Acceptance: existing `annotations_cmd.py` is registered in `main.py`. `annotations list <book_id>` lists all annotations for a book (highlights + notes, grouped by section). `annotations list <book_id> <section_id>` scopes to one section. Create/edit/delete annotations via CLI is a **non-goal** — those are inherently selection-driven UX; CLI users can inspect but not modify.

#### Cluster G — Misc fixes

- **G1. Ask AI button reliably opens the sidebar on AI Chat tab with selection populated. [#7]** Acceptance: on any viewport ≥ 1024px, selecting text and clicking Ask AI within 300ms opens the right sidebar (if closed), switches to the AI Chat tab regardless of previously-active tab, and renders the selection as a quoted-context block in the chat input area. The user can type a question and send. Code recon shows the button emits the correct event but the observed symptom is "nothing happens" — /spec owns triaging the runtime failure mode (likely a missing tab-switch trigger or silent handler error), v1.5 owns shipping a fix that meets the observable criterion above.
- **G2. Books without a cover image get a deterministic title-hash fallback cover. [#8]** Acceptance: any `Book` with no stored cover image renders a generated cover everywhere a cover is shown (library card, book overview header, reader sidebar). The cover is a gradient picked deterministically from a curated 8-gradient palette via `hash(title)`, with the book title centered and auto-wrapped in a serif face. Generated client-side (SVG or canvas). The same title always yields the same cover. No backend asset storage.
- **G3. Library search is debounced at 250ms with a 2-character minimum. [#9]** Acceptance: typing in the library search box does not fire per-keystroke. Search triggers 250ms after the last keystroke and only if the query is ≥ 2 characters. Shorter queries clear results (empty state). A subtle inline spinner appears during the in-flight request. Typing fast enough to generate multiple queued requests cancels the prior in-flight request (AbortController) before issuing the new one.

### Non-Goals (explicit scope cuts)

- **NOT wiring cross-device sync for reader settings**, because localStorage-per-device satisfies the stated "device should remember" intent without introducing settings-sync infrastructure the single-user tool doesn't otherwise need.
- **NOT adding multi-color highlights**, because the current ask is "show highlights inline at all" and single-color ships 80% of the value. Schema reserves a nullable `color` column (see D-decision) so multi-color can ship later without migration pain.
- **NOT AI-seeding section-level tags**, because per-section LLM cost multiplies and the user has stated no explicit ask for that density. Book-level AI tags provide the marquee value at 1/N the cost.
- **NOT adding per-book reader-setting overrides**, because device-global covers the stated use case and invisible-to-user state divergence is a real UX trap.
- **NOT replacing the Settings/Preset picker models with dynamic provider API discovery** (e.g., calling Anthropic's `/v1/models` endpoint), because the CLIs don't expose a list command and direct API enumeration bypasses the "CLI subprocess" abstraction. Config-file-driven curated list is sufficient for v1.5.
- **NOT restructuring reader-settings storage to a reactive Pinia-with-persistence plugin** — the existing `useReaderSettingsStore` with a `localStorage` watcher is enough; architectural rework of the store is a non-goal.
- **NOT implementing shared tags across workspaces / remote sync** — tags are local to the device's DB, same as every other data surface in this tool.
- **NOT adding a unified "per-summarization-job model override"** in the preset picker, because it confuses preset choice with model choice and v1.5's ask was a global default, not per-job selection. Future consideration.
- **NOT building a tag taxonomy management UI** (rename, merge, delete unused) — in v1.5 tags are only editable in-context (per book, per section). Library-wide tag management is a v1.6 candidate once real usage data reveals whether taxonomy hygiene is a felt problem.
- **NOT exposing reader-theme / custom-color / highlight-rendering settings via CLI** — these are browser-local UI state (localStorage) with no meaningful CLI analogue. CLI users reading via `bookcompanion read` get plaintext/markdown output; theme decisions don't apply.
- **NOT supporting annotation create/edit/delete via CLI** — annotations are selection-driven in the UI. CLI registers a read-only `annotations list` (H5) for inspection; write operations stay UI-only.

## User Experience Analysis

### Motivation

- **Job to be done:** "Read a non-fiction book comfortably for 30+ minutes, capture what I want to remember, and come back later and find it." Reading comfort drives theming; "capture" drives notes and tags; "find it later" drives inline highlights and the book overview.
- **Importance/Urgency:** High. Reading is the tool's core loop. Every other surface (search, summaries, chat) exists to support it. If reading is unpleasant, the rest compounds into abandonment.
- **Alternatives:** Kindle, Apple Books, Readwise Reader, a PDF reader. The user is choosing to read here because of the AI summaries + concept extraction — but only if the core reader is pleasant enough that the AI value isn't wasted.

### Friction Points

| Friction Point | Cause | Mitigation |
|---------------|-------|-----------|
| Reading bright white at night | One fixed theme, no Night preset | B1 ships Night as one of six presets |
| "Where is the book summary I just generated?" | `/books/:id/summary` exists but nothing links to it | D1 repurposes `/books/:id` as overview with embedded summary + link to full |
| "My highlight is gone" after navigation | Highlights never render inline | C5 renders inline with pulse-on-scroll (C4) |
| Adding a note feels janky | `window.prompt()` loses selection context, no formatting, no tags | C1 replaces with right-side markdown editor |
| Search feels laggy / characters dropped | Fires per-keystroke, no debounce | G3 adds 250ms debounce + min 2 chars + abort on re-type |
| "Did my summarize-pending click do anything?" | Icon-only button, no loading state, disappears when done | D3 shows "Summarize N of M pending" + spinner + success state |
| "I clicked Ask AI and nothing happened" | Button wired but symptom is invisible | G1 verifies runtime, guarantees tab switch + selection population |
| Cover-less books look broken | Empty tile | G2 auto-generates title-hash gradient cover |

### Satisfaction Signals

- User can open any book in the evening, switch to a dark preset, and read for 30 minutes without touching settings again (B1/B4).
- User can highlight a passage, come back the next day, scroll through the chapter, and see their highlight as a yellow mark in the text (C5).
- User can start typing in search and see results land as they stop typing, with no dropped keystrokes (G3).
- User can add a tag to a book or section and see it render as a chip immediately, across all surfaces (E2/E3).
- User can change the LLM model in Settings and immediately have the next summarization use it (F1).

## Solution Direction

### 1. Theming — preset bundles + curated palettes + WCAG-guarded custom

Six shipped presets, each bundling a paired `bg + fg + font + line-height` combination. Below the preset row in the reader settings popover, two independent color rows: Background (6–7 curated swatches + Custom…) and Text (same). Custom… reveals a color wheel + hex field + live contrast-ratio readout with WCAG AA/AAA badges. Warning but not blocking on low-contrast combinations. All state in `localStorage` under a single `readerSettings` key. The existing `useReaderSettingsStore` extends to include the new color state; serialization is a JSON dump on change.

### 2. Highlights inline — `<mark>` wrapping with theme-aware tint

A shared "apply highlights to rendered HTML" post-processing step runs after markdown-it + DOMPurify, walks the rendered DOM for text ranges matching stored highlights, and wraps them in `<mark data-highlight-id="{id}">`. Theme CSS sets `mark { background: var(--highlight-bg); }` where `--highlight-bg` varies per preset (full-sat yellow on light presets, desaturated ochre on Night). Sidebar click → `document.getElementById(...)` + `scrollIntoView({ behavior: 'smooth', block: 'center' })` + 1.5s CSS keyframe pulse. Same processor runs on both Original and Summary tabs.

### 3. Annotations scope + scroll-to-source

The Annotations tab filters stored annotations by `section_id === currentSectionId` by default. A "Show N annotations from other chapters" row below expands into section-grouped accordions. Click any annotation → if same section, scroll in place; if different section, `router.push('/books/:id/sections/:sid#highlight-:aid')` and on mount scroll to the anchor. State: `otherChaptersExpanded = false` resets on section navigation.

### 4. Note input panel

A new right-sidebar state `noteComposition = { mode: 'create' | 'edit', selection, existingAnnotationId? }`. When Ask AI shares the sidebar, the panel is a dedicated state. Uses the existing sidebar layout — no new modal shell. Markdown editor is a textarea with a toggle to switch to a live `MarkdownRenderer` preview (same renderer the reader uses). Tags use a chip input with autocomplete from existing tags.

### 5. Book overview page (`/books/:id`)

Router change: `path: '/books/:id'` now points to a new `BookOverviewView.vue` (the existing `BookDetailView.vue` continues to own `/books/:id/sections/:sid`). Layout:

```
┌─────────────────────────────────────────────┐
│ [cover] Title                               │
│         Author · Format · N/M summarized    │
│         ▓▓▓▓▓▓▓░░░░ progress                 │
│         [Resume reading]                    │
├─────────────────────────────────────────────┤
│ Book summary                                │
│ (rendered markdown, or Generate CTA)        │
│                    [View full summary →]    │
├─────────────────────────────────────────────┤
│ Sections               [Summarize N pending]│
│  ✓ 1. Introduction  · [#foundation]         │
│  ✓ 2. Competition   · [#strategy]           │
│  — 3. Value         · [+]                   │
│  ⚠ 4. Creating...   · [+]                   │
└─────────────────────────────────────────────┘
```

### 6. Tags — polymorphic association + AI seeding

New Alembic migration re-adds `tags` and `taggables` tables (unique case-insensitive name, optional color, polymorphic type/id). API: `GET/POST /api/v1/books/:id/tags`, `DELETE /api/v1/books/:id/tags/:tagId`, same shape for sections. Summarizer service is extended: after a book-level summary completes, a follow-up LLM call requests 3–5 tags (small cheap prompt) and attaches them to the SSE `processing_completed` event as `suggested_tags`. The overview page shows them as pending chips until the user accepts.

### 7. LLM model selector

New file `backend/app/config/models.yaml`:

```yaml
claude:
  - id: sonnet
    label: Claude Sonnet (default, balanced)
  - id: opus
    label: Claude Opus (highest quality, slower)
  - id: haiku
    label: Claude Haiku (fastest, lowest cost)
codex:
```

### 8. Bug fixes

- **A1 (summary images):** in `SummarizerService`, just before persisting `summary_md`, call `from_placeholder(summary_md, filename_to_image_id)` the same way `content_md` is rewritten. Data migration walks existing summaries and rewrites any remaining placeholders.
- **A2 (blockquotes):** new post-processing step in `EpubParser` (and PDF parser for symmetry) that runs after `markdownify`: collapse nested `> >` / `> > >` chains to `>`, strip `---` that appears as first or last line inside a blockquote. Migration re-normalizes existing `content_md`.
- **D2 (next/prev routing):** in `useReaderStore.navigateSection`, replace direct `loadSection(...)` with `router.push({ name: 'section-detail', params: { id, sectionId } })`. The existing `loadFromRoute` watcher handles the rest.
- **G3 (search debounce):** replace per-keystroke fetch in the library search with a `useDebounceFn` wrapper (250ms) + min-length guard + AbortController for in-flight cancellation.

## User Journeys

### Primary Journey: Evening reading session with theming

1. User opens the reader around 9pm. Bright white hurts; opens the reader settings popover.
2. Sees six preset cards. Clicks Night. Article, summary tab, sidebar, toolbar all transition to a dark theme instantly. Night is saved automatically (preset change → auto-persist).
3. Reads for 15 minutes. Decides the text is slightly too blue; opens Text Color row, clicks Custom…, types `#d8d3c8` in the hex field. Contrast readout shows 11.2:1 with a green "AAA" badge. The reader **previews** the new color live, but a "Save & Apply as Custom" CTA appears above the preset row signaling unsaved state.
4. User clicks Save & Apply. The Night preset card deselects; a "Custom" indicator becomes the active state; the custom bg/fg/font/size combination is persisted to localStorage.
5. Closes the popover. Continues reading. Closes the tab.
6. Next evening, reopens the tool. Opens the same book (or any book). The Custom theme is already applied. (Had the user closed the popover without clicking Save & Apply in step 4, the reader would have reverted to Night on reload.)

### Primary Journey: Highlight and find it later

1. User selects a passage in section 4. Floating toolbar shows Highlight / Note / Copy / Ask AI.
2. Clicks Highlight. Selection clears; the passage now has a yellow background mark (C5) — the theme-aware tint shows correctly on whatever preset is active.
3. Navigates to section 7 via Next (URL updates — D2).
4. Later, opens the Annotations tab. Sees section 7's annotations (empty). Clicks "Show 3 annotations from other chapters". Section 4 expands. Clicks the section-4 highlight row.
5. Reader navigates to section 4 (route change), smooth-scrolls to the highlighted passage, which pulses once so the user can spot it.

### Primary Journey: Tag a book with AI-suggested tags

1. User finishes generating a book-level summary on the Porter book.
2. Book overview page auto-refreshes. Under the summary zone, a "Suggested tags" pill group appears: [strategy ✓/✗] [competitive-advantage ✓/✗] [porter ✓/✗] [business-frameworks ✓/✗] [value-chain ✓/✗].
3. User clicks ✓ on strategy, competitive-advantage, porter. Clicks ✗ on the other two.
4. The three accepted tags move up into the book's tag row. "Suggested tags" group disappears.
5. User opens section 1. Adds a manual section tag "foundations" via the + button in the reader header.
6. Back on the library view, clicks the "strategy" chip on the Porter card. Library filters to books tagged strategy.

### Error / Edge Cases

| Scenario | Condition | Expected Behavior |
|---|---|---|
| User sets custom colors below WCAG AA | Contrast < 4.5:1 | Warn badge shown; apply succeeds; no block |
| User types < 2 chars in search | Query length 1 | No request; prior results cleared; empty state |
| User switches model mid-job | LLM.model changed while a summarization job is running | Running job completes with old model; next job uses new model |
| Book with no cover + no title (edge case) | Title empty | Fallback cover renders gradient only, no text; log warning |
| Suggested tag already exists as user-added | Dedup on save | Pending chip is pre-accepted (✓); user can ✗ to reject |
| User accepts 0 suggested tags | All ✗ | Suggested group disappears; book tags unchanged |
| Nested blockquote normalization hits 3 levels | `> > > x` in source | Collapsed to `> x` (all non-top levels flattened) |
| Highlights toggle off mid-read | User flips "Show inline" | `<mark>` styling removed without DOM re-render churn; toggle back on restores |
| Model specified via Custom… is rejected by CLI | Subprocess exits non-zero | Stderr truncated + shown on section Summary tab + banner on overview; Settings dropdown reverts to prior selection on retry |
| `/books/:id` visited before overview ships | Route already exists pointing to reader | Migration ensures the new overview route supersedes the reader route; reader stays only at `/sections/:sid` |

### Empty States

| Scenario | Condition | Expected Behavior |
|---|---|---|
| Book with no summary, no sections summarized | Fresh upload | Overview summary zone shows Generate CTA + "Why summarize?" explainer; TOC shows all `—` pending; "Summarize N of N pending" button prominent |
| Book with 0 annotations | Nothing highlighted or noted yet | Annotations tab shows "No annotations yet. Select text to highlight or add a note." |
| Book with 0 tags, no summary yet | Pre-summary | Book overview tag row shows "+ Add tag" affordance only; no suggested-tags group |
| No LLM provider detected | Fresh install, no CLI on PATH | Settings LLM section shows "No provider detected" banner; Model dropdown disabled with "Install Claude or Codex CLI" help link |

## Design Decisions

| # | Decision | Options Considered | Rationale |
|---|---|---|---|
| D1 | Fix summary-image placeholders at the backend save path, not the FE render path | (a) FE runs `from_placeholder` at render time; (b) BE rewrites on save (chosen); (c) BE rewrites on every read | Save-path fix is the canonical home for that transformation — `content_md` already does it there. (a) duplicates the rewrite rule on the client; (c) burns CPU per read. Same normalization rule, same call site. |
| D2 | Collapse nested blockquotes + strip in-quote HR markers at parse time | (a) CSS-only flatten (border on outermost only); (b) parser normalization (chosen); (c) both | Parser fix produces clean stored markdown that's correct for any future renderer, exporter, or LLM prompt. (a) leaves garbage in the data layer. (c) is belt-and-suspenders but adds maintenance on a single observation; revisit only if a new case breaks. |
| D3 | Ship six named theme presets (not three, not nine) | (a) 3 (Light/Sepia/Dark); (b) 6 (chosen); (c) 9+ with user-saved themes | Six covers the Apple Books inventory (Original/Quiet/Paper/Bold/Calm/Focus) well — enough variety without decision paralysis. Nine presets means naming them becomes absurd; the separate color pickers pick up the flexibility. |
| D4 | Custom hex is allowed but gated through a WCAG contrast readout, not blocked | (a) curated palette only; (b) unconstrained hex; (c) hex with warning (chosen); (d) hex with hard block below AA | User explicitly asked for hex input. Blocking is paternalistic for a single-user tool. Warn-but-allow respects autonomy while making the tradeoff visible. |
| D5 | Reader settings stored device-global in localStorage, one key, no per-book overrides | (a) global (chosen); (b) per-book; (c) global + per-book overrides | User said "device should store." Per-book introduces invisible divergence; no evidence of real need. Single key simplifies clear + export + debug. |
| D6 | Single highlight color (theme-aware yellow) now, schema reserves `color` nullable for later | (a) single color, no reserve; (b) single color, nullable `color` column (chosen); (c) multi-color v1 | Ships the 80% value now without migration pain later. The `color` column costs nothing and unlocks multi-color v1.6 without a schema break. |
| D7 | Annotations tab default = current section only; toggle expands other chapters grouped | (a) all annotations flat; (b) current-first with divider; (c) current-only with toggle (chosen); (d) current-only, other-chapters via route | User's explicit ask. (d) is more purist but adds nav cost for cross-section comparisons that matter when writing summaries. |
| D8 | Note input is a right-sidebar panel, not a modal or inline popover | (a) modal; (b) inline popover; (c) right-sidebar panel (chosen) | Matches Hypothesis/Readwise. Preserves reading context, allows markdown editing + tag chip input without cramping. Modal is intrusive; inline popover is too small for tags + preview. |
| D9 | Book overview at `/books/:id` replaces the reader's old claim on that route | (a) add `/books/:id/home` (chosen by URL not chosen by semantics); (b) repurpose `/books/:id` (chosen); (c) add a nav link only | Breadcrumb semantics and library-click semantics both expect "the book" to land on a book surface. Repurposing is cheap (one router change + new view). New route is a URL hack that makes the click target less meaningful. |
| D10 | Tags schema: polymorphic Taggable join, not separate book_tags + section_tags | (a) two join tables (simpler schema); (b) polymorphic single table (chosen) | Polymorphic matches the stubbed schema already in the ORM; two-table requires deleting the existing model. Polymorphic supports future tag targets (annotations, concepts) without new tables. |
| D11 | AI seeds BOOK tags only, at book-summary completion, with accept/reject flow | (a) book auto-apply; (b) book accept/reject (chosen); (c) book + section auto-apply; (d) no AI seeding | Accept/reject gives the user control on first exposure, which matters for trust. Section-level seeding is future scope — per-section LLM call cost adds up and the first round of user signal should come from book-only. |
| D12 | LLM model list lives in `backend/app/config/models.yaml`, loaded at startup | (a) hardcoded in the Python file; (b) YAML config (chosen); (c) dynamic provider API call | User's explicit preference. YAML lets the list grow across releases without a code change. Dynamic API discovery bypasses the CLI subprocess abstraction — violates the design invariant. |
| D13 | Model selector in Settings is the single override; no per-summarization-job override | (a) Settings-only (chosen); (b) also per-job in preset picker; (c) also per-book | Per-job conflates preset choice with model choice; per-book is a v1.6 consideration. Settings-only keeps the mental model clean. |
| D14 | Cover fallback is client-generated from `hash(title)` against a curated 8-gradient palette | (a) solid single color; (b) title-hash gradient (chosen); (c) Unsplash/stock-image fetch | Gradient palette gives visual variety while remaining tasteful; deterministic hash keeps the cover stable across renders. Stock-image fetch introduces a network dependency for a personal tool and raises legal/caching complexity. |
| D15 | Debounce at 250ms with min 2 chars, plus AbortController on re-type | (a) 150ms/3 chars; (b) 250ms/2 chars (chosen); (c) 400ms/2 chars | 250ms is the industry sweet-spot (GitHub, Algolia, Google instant). Two chars matches user ask. AbortController prevents stale-result flicker on fast typers. |
| D16 | Non-preset customizations require explicit Save & Apply to persist; live preview without save reverts on reload | (a) auto-save on any change (Readwise pattern); (b) auto-save color only, font/size stay in preset; (c) explicit Save & Apply for all non-preset deviations (chosen) | User ask — persistence must be intentional, not incidental. Prevents accidental one-off tweaks from overwriting a preferred preset. One Custom slot (overwrite on save) keeps the mental model simple; named multi-theme support is a v1.6 consideration. |
| D17 | "Summarize pending" button lives only on the book overview, not in the reader header | (a) both places with shared store (earlier draft); (b) overview-only (chosen); (c) reader-only | Single source of truth eliminates sync complexity. Overview is the natural home for book-level batch operations. Reader users reach it via breadcrumb / new D1 routing — one extra click is acceptable for a rare action. |
| D18 | CLI `tag suggest` is a READ of stored suggestions, not an on-demand LLM call | (a) CLI triggers suggestion generation; (b) CLI reads stored suggestions (chosen); (c) no CLI tag-suggest at all | Keeps UI and CLI in exact parity on the one-shot model (E4 / D11). Avoids two divergent code paths for suggestion generation. If user wants new suggestions, they regenerate the book summary — which in v1.5 intentionally does NOT re-suggest (F2), making the non-generation stance consistent. |
