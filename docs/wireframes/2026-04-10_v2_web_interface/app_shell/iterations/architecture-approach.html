Architecture Approach

How should the web app be structured at the highest level?

A

Reader-Centric (Recommended)

The web app is primarily a reading and exploration interface. Books are the center of gravity — everything flows from selecting a book. Upload, summarization, and eval are actions you take on a book, not separate workflows.


Strengths

  • Natural mental model: "I want to read/explore my books"
  • Minimal navigation — most actions accessible from book context
  • Search and concepts are cross-cutting layers, not separate pages
  • Mirrors how you actually use the CLI (add → summarize → read → search)

Trade-offs

  • Cross-book features (global search, concept graph) need careful placement
  • Book detail page becomes feature-heavy — needs good information architecture
B

Dashboard-Centric

Home is a dashboard with activity feed, recent books, processing status, and quick actions. Each major feature (Library, Search, Concepts, Annotations) is a top-level page. More like a knowledge management platform.


Strengths

  • Each feature area gets dedicated space and attention
  • Dashboard provides at-a-glance overview of library state
  • Scales well if features keep growing

Trade-offs

  • More navigation, more clicking between pages
  • Dashboard can feel empty with a small library
  • Over-engineered for a personal tool — feels like enterprise software
C

Command-Palette-First

Minimal chrome. The app opens to your library. Cmd+K palette is the primary navigation — search books, jump to sections, trigger actions. Pages are minimal; the palette is the power interface. Think Raycast/Alfred for your book library.


Strengths

  • Fastest navigation for power users
  • Minimal UI surface area to build and maintain
  • Closest to the CLI mental model

Trade-offs

  • Discovery problem — new users won't know what commands exist
  • Some features (annotation browsing, concept graph) need visual space
  • Harder to show processing progress and status
